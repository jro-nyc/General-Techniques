Avis launched their AngularJS Single Page Application (SPA) the week of 9/26/2016. Third party page manipulation is more difficult when the client is using Angular. The purpose of this page is to provide guidance and act as documentation center for information gathered about working with Avis and their AngularJS SPA.

Setup

There is a throttle on their homepage allowing traffic to the Angular site as they test it and monitor its performance. The throttle is managed using cookies.

Follow the below steps to enter the correct Avis homepage.

Steps

1) Open a new tab/window and go to this URL: https://www.avis.com/

2) Run this code from the browser developer tool bar console:

  1. // Drop a cookie named "datacenter" with the value "newboulder" and save that cookie for 30 days in the *.avis.com domain
  2. optimost.SC("datacenter", "newboulder", 2592000, optimost.SLD());  

3) Type https://www.avis.com/ into the navigation bar, and hit enter. 

Warning:
If you were put into the old version of the website, refreshing the page will not take you to the new site after completing the steps above. You must navigate to the root path of avis.com. 

Optimost Implementation

The Optimost implementation used by Avis is non-standard on their AngularJS website due to the unique structure of AngularJS websites. A brief explanation of the issues and current code structure is required to understand how the Optimost code is capable of making Angular implementation changes.

Issues

Optimost encountered issues while integrating with Avis' Angular site. Each issue below issues (and resolution) dictated changes creating the page load flow the site has currently.

Late Optimost Load

In the first attempts of integration, Avis loaded made the initial call to the Optimost server at the end of the body element, right after their implementation code executed. This was causing the Optimost test code to execute after the page had loaded. Optimost must fire before the Angular framework processes the implementation code in order to make changes. 

Resolution

The Optimost page selector code was moved to the head block, and is fired asynchronously. The code is located in optimost.js .

Execution Timing Issues

The Angular framework begins processing immediately after the JavaScript file containing the Avis Angular implementation code fires. Making changes to the Angular implementation is timing sensitive and, during initial testing, we were unable to reliably execute after their implementation code, but before the Angular framework begins processing. 

Resolution

We requested code be added to their JavaScript file that looks for an Optimost Hook function to be defined, and executes it if it is defined to guarantee our code is executed after the Avis code fires but before Angular processes it. All tests must define optimost.optimostHook in the challenger.

Page Flow

Understanding how the page loads is required to understand when our changes are being made. The below mock-up of the how the page loads within the scope of AngularJS, Avis implementation code, and the Optimost code.

The current load order allows Optimost to overwrite any Angular controllers/services/factories/etc... that may need to be overwritten to provide/remove/modify page functionality for a test. The overwrite is possible because Angular only allows one definition of a controller, service, factory, provider, value, and constant.

Modifying Angular Implementation Code

The Angular framework changes how Optimost modifies and changes content on the page. In a normal web page, the HTML DOM contains all relevant data and changes required in JavaScript were linked to HTML elements. Basic JavaScript and DOM changes could rework an entire page. The introduction of the Angular framework changed the standard funnel into a Single Page Application (SPA). Angular acts as a middle man between written code, and the DOM. This changes the dynamic in our tests. We are no longer modifying the DOM and modularized JavaScript functions; we are modifying underlying Angular implementations to reflect the changes we need to see in the DOM. The task requires reverse engineering of client code, and a strong understanding of how Angular interacts with the DOM and the written client implementation to make the required changes without causing issues in other areas of the client's SPA.

DOM Modifications

The DOM is, generally, left to Angular framework to modify, as needed, based on the implementation details. Angular uses client create HTML templates (or partial templates) and information from the model and controller to render the dynamic content the user sees in the browser. 

Here's an example from the Avis car reservation page (https://www.avis.com/en/reservation#/vehicles ).

The template is HTML with Angular specific properties defined. The properties usually contain references to implementation specific variables and functions.

Updating the stored templates proves difficult, and we are still working on accomplishing such a task as Avis does not follow the standard Angular practices we experience with other clients.

Debugging Angular Implementation Code

Optimost is capable of making changes to the Avis' implementation code. To do so effectively, debugging is required. The process of finding where changes need to be made varies, but the below steps should aid in the efforts most times.

Identfying Important Variables

Most client requests have a targeted location, for example. Avis may ask Optimost to add a vehicle type to their dropdown menu.

This requires making changes to the variables stored within the Angular scope. We use the variables mentioned in the Angular specific HTML properties to help narrow the scope of our search.
Here is the HTML of the darkened area above:

  1. <!-- ngRepeat: vehValue in vm.response.carGroupList -->
  2. <li class="col-sm-6 list-group-item vehicleTypeFilter" vehiclecode="all" vehiclecount="23" ng-repeat="vehValue in vm.response.carGroupList" ng-click="!(vm.getCarCountByClass(vehValue.cargroup)===0) ? vm.filterVehicle(vehValue,false) : ''">  
  3.     <div class="col-sm-12 vehListRule hover" ng-class="{'disabled-because-noveh disabled': vm.getCarCountByClass(vehValue.cargroup)===0,'hover' : vehValue.code===vm.pageControl.displayedVehicleType &amp;&amp; vm.getCarCountByClass(vehValue.cargroup)!==0}">  
  4.         <img ng-attr-src="{{vehValue.image}}" alt="All Vehicles" src="/content/dam/avis/na/us/common/cars/trucks-vans.png/jcr:content/renditions/cq5dam.thumbnail.151.100.png">  
  5.         <span>All Vehicles (23)</span>  
  6.     </div>  
  7. </li>  
  8. <!-- end ngRepeat: vehValue in vm.response.carGroupList -->

There are several starting locations, but since we need to add an entire section, we should start searching through the sources for the variable vm.response.carGroupList. The reasoning behind this is the comments above were generated by the Angular Framework. That means we know it exists, and, with no prior information or knowledge of the system, it's a solid starting location.

Before we even open the sources panel in the developer tools, we know that vm.response.carGroupList contains objects used to create elements in the drop down. The real challenge is learning where that content comes from and understanding how it's built because it may be used in many locations, and we can't cause those areas to fail when we add our new area.

Readability of Clientlib File

Avis' Angular implementation file (https://www.avis.com/etc/designs/platform/clientlib.min.1.0.9.js ) is compressed to reduce the file size. The compression makes debugging the implementation code impossibly difficult. The non-compressed file can be found by removing the .min (https://www.avis.com/etc/designs/platform/clientlib.1.0.9.js ) from the file name. The debugging process is significantly easier to do with a human-readable code set, so we use Fiddler  to help fetch the readable version when the call is made for the compressed version.

Note:
Fiddler4, can be downloaded here .
Please review the Fiddler Setup page before continuing.

Using Fiddler

The following steps will force the request of the minified code to fetch the human-readable code from the Avis servers.

  1. Open Fiddler4 (if you do not have Fiddler installed, please review the setup page).
  2. Click on the FiddlerScript tab.
  3. Click the dropdown and select OnBeforeRequest.
  4. At the end of the function, add the following code:
    1. // While in the www. Avis domain, change all requests for clientlib.min.*.*.*.js to clientlib.*.*.*.js  
    2. if (oSession.HostnameIs("www.avis.com") && oSession.url.IndexOf("clientlib.min.") > -1) {  
    3.     oSession.url = oSession.url.replace("clientlib.min", "clientlib");  
    4. }  
  5. Click the Save Script button.

    Warning:
    The keyboard shortcut CTRL+S will not save the file.

    You must click the Save Script button or use the shortcut ALT+i.

There are several ways to confirm the changes are working, but the easiest way is to review the file's content in the developer tools' Sources tab.

Note:
The name of the file in chrome will not change. It will still have the .min in the file name, even if the content within is the readable code.

This is because Fiddler does not interact with the browser when changing the request, so the browser is unaware of modifications.

Caution:
If you are still seeing the compressed file, be sure to clear your cache, or force the page to reload using CTRL+F5 (forces a cache refresh).

A cached version cannot be modified by Fiddler.

Navigating The Implementation Code

A large amount of time is spent going through the implementation code, and stepping through it one line at a time once you have a better understanding of where your changes need to be made. This process is made easier by using the developer tools on the browser. Going forward, my examples and images will come from the Chrome developer tools. Continuing with the above example, we take our variable and use the search all files tool to find all matching locations.

Begin looking for functionality at the listed locations. If the location doesn't have the information needed, start eliminating the parent objects while searching eg: when vm.response.carGroupList doesn't work, trying response.carGroupList. If that doesn't give enough details in the code, move to carGroupList.

Pretty Print

 As you begin reading through the code, you may want to improve the displayed layout. Below the code, but above the search box in the developers tools is a Pretty print button. Once clicked it will correct the code layout to the best of its ability (which is pretty good).

Now, if you search for the term response.carGroupList again, you will note it finds two occurences: the original (as shown above) and one in a file named the same, but with :formatted appended to the file path. They are the same file, but the :formatted link is in the now properly laid out file.

Avoiding Unnecessary Rabbit Holes

As you get more general in your search (from the above example), we haven't found anything useful in the search until we looked for carGroupList. However, there are 17 matches, and to go through each one in detail would take too long. Identifying irrelevant locations is crucial to rapid development and quick (relative term) turnarounds. Here are some suggestions/pitfalls to look out for while trying to find relevant usage of your search term:

  • Located in unrelated function names
    When you find the search term, first identify the name of the function it is located in. If the function is not relevant to your main objective, you can ignore that instance of the term and quickly move to the next one.
    Example:
    We find carGroupList in a function called displaySoldoutInstruction. We aren't worried about functionality at the bottom of the page, and the task describe, while related to car types, is not relevant because we need to find out how they are created, not how they are used (at least not yet).
  • Losing variable focus
    When looking through code, it is easy to get caught up trying to determine if a function using the term you are searching for is relevant. Though sometimes required, it is not something you want to start with, but, rather, a step taken once how the variable is created. It's trying to jump ahead without proper information.
    Example:
    The function filterVehicle seems to take the variable carGroupList as a parameter. You would be losing variable focus if you decided to determine the purpose and process of the function. That step is only helpful if you have a strong understanding of what the carGroupList variable represents. Delving into the function without a firm understanding of the variable forces you to learn what the variable represents while simultaneously learning the function's purpose. The spilt in concentration generally leads to confusing, or continuous focus loss.
  • Over investing
    When searching through the code, you may follow a variable chain to a dead end. Trying to piece your way back and recover is difficult (unless the chain is very short). It is usually easier to go back to your original search term, refocus, and start from the top.
     

Pattern Identification and Code Associations

After reviewing the search results, you begin to find patterns to where and how the search term is being used. Continuing with our above example, we noted that carGroupList (and functions that contain carGroupList) is located several times in the function dataProcess. This gives us a location to begin a more in-depth search on how carGroupList is used.

The beginning of the function proves promising, as it has the original parent objects from our first search term: vm.response.carGroupList.

This means that we can view the variable data (at least in this function) and the vm.response as the same thing. This further shows we are in the right place because of the multiple data.carGroupList references we see in the function.

There is strong evidence that more in-depth review of this function will further bear fruit in our search for understanding.

Breakpoints and Analysis

Adding a breakpoint allows you to stop the code processing and view all the variables at their current values in the code where the breakpoint was placed. This will shed light on carGroupList. Adding a breakpoint is easy, all you need to do is click on the row number to add it.

Once the page is reloaded, the page will lock, and the debugger will provide data on the scope variables, and call stack by default. You can add variables to watch in the top right of the screen.

A review of the variable data.carGroupList shows that it is an array and each object is comprised of the following items:

  • CarList - An array of varying size
  • carModel - An array of varying size
  • cargroup - An array of varying size
  • code - A string that seems to be a shortened version of the description
  • description - self-explanatory; however, this text shown (in part) for the vehicle types on the actual page
  • image - the path for the image to display (confirmed by checking the image from the page using another tab)

Scaling the Call Stack 

We have determined that the data object is closely associated with the carGroupList variable. After a brief review of the data object, we realize the data object is populated before it reaches this function. It's time to figure out where the data object is created/populated. We'll do this by using the call stack while debugging the above example. The Call Stack provides the order of calls made in JavaScript. Each line in the call stack is where the execution will begin once finished with the above section.

In our example, the dataProcess is called from an anonymous function.

To view that area, click the anonymous function section within call stack. This takes us to a the anonymous function within a then call after a call to the function getData.

The name of the function is exactly what we trying to define. It's time to review the function getData. There are several ways to get to the function, but the least error prone method is to add a breakpoint on the line containing the function we want to review. To review the function getData, we add a breakpoint on line 14808 and refresh the page.

Step Through Learning

The process of stepping through is great for observing exactly what happens as code is executed on the page. This process involves using the step tools provided by the developer tools.

Here is a cheat sheet for understanding and using the tool:

Now that our breakpoint has halted code on the actual line calling getData, we need to go into the function, and see find out where the data object is being created/populated. After going into the function the code is probably foreign (unless you have some serious AngularJS experience) to you. This means the first step in this function is to understand what is going on.

Usually, this means getting on Google  (or your preferred search engine) and doing some searches to figure out what $q means, and what defer() does. Luckily, this is a tutorial, so you get to skip the hour or so of learning about $q from Google. $q is an AngularJS provider that helps you run functions asynchronously, then use their return values when they are done processing. It is used when you need to make an API call to get required data for the code. Here are the key points you need to understand $q and promises:

  • The function call defer() creates something called a promise. A promise represents the future object to be delivered while waiting for the information required to complete execution to become available.
  • The returned object deferred.promise is the promise mentioned above. 
  • The then function is called on the deffered.promise object passed back in the return statement. then defines what happens after you get the information required. It allows you to begin processing with the promised object.

With our newly acquired knowledge, and a bit of digging into the other functions in the code, we now have an idea of what is going on in this function:

The reservationModel is pulled from storage, if available, a promise is created, and the data object is pulled from storage, if available.
If the data is not stored, the server API is called using an AJAX call with a promise returned containing the required information to populate the data object.
However, if the data is available, the data object is processed, and the promise is resolved, meaning there is no need to wait on anything and to continue to the then call.


This analysis shows that data is populated by the web server first. This means that data.carGroupList is also populated by the server. Now, with a firm understanding of where the variable is created/populated, we can start learning about what each variable in the objects of the carGroupList array are for, and how changes to them affect the page.

Next Steps

The above provides the foundation of walking through the application. The next steps are to understand how the content interacts. 

The best way to understand how content interacts in an application, is to, with a firm understanding of the content's origin, start making modifications to the code that aid in our goal of creating a new vehichle type. In our example, the next steps would be to change an existing vehicle type section into the new one we want to create. These steps are standard debugger steps, and will not be covered in the Angular specific document.

The task requires reverse engineering of client code, and a strong understanding of how Angular interacts with the DOM and the written client implementation to make the required changes without causing issues in other areas of the client's SPA.

DOM Modifications

The DOM is, generally, left to Angular framework to modify, as needed, based on the implementation details. Angular uses client create HTML templates (or partial templates) and information from the model and controller to render the dynamic content the user sees in the browser. 

Here's an example from the Avis car reservation page (https://www.avis.com/en/reservation#/vehicles ).

The template is HTML with Angular specific properties defined. The properties usually contain references to implementation specific variables and functions.

Updating the stored templates proves difficult, and we are still working on accomplishing such a task as Avis does not follow the standard Angular practices we experience with other clients.

Readability of Clientlib File

Avis' Angular implementation file (https://www.avis.com/etc/designs/platform/clientlib.min.1.0.9.js ) is compressed to reduce the file size. The compression makes debugging the implementation code impossibly difficult. The non-compressed file can be found by removing the .min (https://www.avis.com/etc/designs/platform/clientlib.1.0.9.js ) from the file name. The debugging process is significantly easier to do with a human-readable code set, so we use Fiddler  to help fetch the readable version when the call is made for the compressed version.

Note:
Fiddler4, can be downloaded here .
Please review the Fiddler Setup page before continuing.

Using Fiddler

The following steps will force the request of the minified code to fetch the human-readable code from the Avis servers.

  1. Open Fiddler4 (if you do not have Fiddler installed, please review the setup page).
  2. Click on the FiddlerScript tab.
  3. Click the dropdown and select OnBeforeRequest.
  4. At the end of the function, add the following code:
    1. // While in the www. Avis domain, change all requests for clientlib.min.*.*.*.js to clientlib.*.*.*.js  
    2. if (oSession.HostnameIs("www.avis.com") && oSession.url.IndexOf("clientlib.min.") > -1) {  
    3.     oSession.url = oSession.url.replace("clientlib.min", "clientlib");  
    4. }  
  5. Click the Save Script button.

    Warning:
    The keyboard shortcut CTRL+S will not save the file.

    You must click the Save Script button or use the shortcut ALT+i.

There are several ways to confirm the changes are working, but the easiest way is to review the file's content in the developer tools' Sources tab.

Note:
The name of the file in chrome will not change. It will still have the .min in the file name, even if the content within is the readable code.

This is because Fiddler does not interact with the browser when changing the request, so the browser is unaware of modifications.

Caution:
If you are still seeing the compressed file, be sure to clear your cache, or force the page to reload using CTRL+F5 (forces a cache refresh).

A cached version cannot be modified by Fiddler.

Avoiding Unnecessary Rabbit Holes

As you get more general in your search (from the above example), we haven't found anything useful in the search until we looked for carGroupList. However, there are 17 matches, and to go through each one in detail would take too long. Identifying irrelevant locations is crucial to rapid development and quick (relative term) turnarounds. Here are some suggestions/pitfalls to look out for while trying to find relevant usage of your search term:

  • Located in unrelated function names
    When you find the search term, first identify the name of the function it is located in. If the function is not relevant to your main objective, you can ignore that instance of the term and quickly move to the next one.
    Example:
    We find carGroupList in a function called displaySoldoutInstruction. We aren't worried about functionality at the bottom of the page, and the task describe, while related to car types, is not relevant because we need to find out how they are created, not how they are used (at least not yet).
  • Losing variable focus
    When looking through code, it is easy to get caught up trying to determine if a function using the term you are searching for is relevant. Though sometimes required, it is not something you want to start with, but, rather, a step taken once how the variable is created. It's trying to jump ahead without proper information.
    Example:
    The function filterVehicle seems to take the variable carGroupList as a parameter. You would be losing variable focus if you decided to determine the purpose and process of the function. That step is only helpful if you have a strong understanding of what the carGroupList variable represents. Delving into the function without a firm understanding of the variable forces you to learn what the variable represents while simultaneously learning the function's purpose. The spilt in concentration generally leads to confusing, or continuous focus loss.
  • Over investing
    When searching through the code, you may follow a variable chain to a dead end. Trying to piece your way back and recover is difficult (unless the chain is very short). It is usually easier to go back to your original search term, refocus, and start from the top.

Step Through Learning

The process of stepping through is great for observing exactly what happens as code is executed on the page. This process involves using the step tools provided by the developer tools.

Here is a cheat sheet for understanding and using the tool:

Now that our breakpoint has halted code on the actual line calling getData, we need to go into the function, and see find out where the data object is being created/populated. After going into the function the code is probably foreign (unless you have some serious AngularJS experience) to you. This means the first step in this function is to understand what is going on.

Usually, this means getting on Google  (or your preferred search engine) and doing some searches to figure out what $q means, and what defer() does. Luckily, this is a tutorial, so you get to skip the hour or so of learning about $q from Google. $q is an AngularJS provider that helps you run functions asynchronously, then use their return values when they are done processing. It is used when you need to make an API call to get required data for the code. Here are the key points you need to understand $q and promises:

  • The function call defer() creates something called a promise. A promise represents the future object to be delivered while waiting for the information required to complete execution to become available.
  • The returned object deferred.promise is the promise mentioned above. 
  • The then function is called on the deffered.promise object passed back in the return statement. then defines what happens after you get the information required. It allows you to begin processing with the promised object.

With our newly acquired knowledge, and a bit of digging into the other functions in the code, we now have an idea of what is going on in this function:

The reservationModel is pulled from storage, if available, a promise is created, and the data object is pulled from storage, if available.
If the data is not stored, the server API is called using an AJAX call with a promise returned containing the required information to populate the data object.
However, if the data is available, the data object is processed, and the promise is resolved, meaning there is no need to wait on anything and to continue to the then call.


This analysis shows that data is populated by the web server first. This means that data.carGroupList is also populated by the server. Now, with a firm understanding of where the variable is created/populated, we can start learning about what each variable in the objects of the carGroupList array are for, and how changes to them affect the page.
