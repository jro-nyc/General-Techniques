Tutorial Prerequisites

In this tutorial, I assume you are not only familiar with some commonly used web tools, but that you use them on a regular basis. In order to do most of the testing/debugging reguired using Optimost, you will need to use a Browser that sas built-in or add-on functionality to view:

  1. The DOM of a web page (being able to edit/modify it is a plus, as is being able to see the CSS)
  2. All the external calls associated with a web page load.1
  3. The flow of Network packets associated with a browser and the pages you go to. 1
  4. Cookies and their relevant information.

For all of these, I use Firefox with Firebug and Livehttpheaders as add-ons. Other browsers have similar tools, but I find the Firefox add-ons to be the best. You may hear references to these tools in this tutorial, so familiarize yourself with them.

1Note that although #2 and #3 are very similar, only #3 can help you debug what happens when you perform an "event" on a page that takes you to another page.

OVERVIEW - The Philosophy of Testing

Optimost is a testing platform. The goal of testing is to get information about the behavior of visitors to a website so that our customers can improve the user experience. We want to find out what visitors like AND what they dislike. We may test visitors as one homogeneous groups - what we call the All Visitors "Persona" - or we may be able to separate the traffic into heterogeneous groups - say different "Personas" based on Geography, demographics, or how they got to the website (whether it was organic search or they clicked on advertisement). So when we test a page, we pick a Key Performance Indicator - KPI - as the success point, and see how visitors that land on our test page "perform" compared to each other based on that KPI. The KPI may differ based on what the customer considers to be the success - it may be clicking on some offer, enrolling in a newsletter, or buying a product.

The key component for accurate measurement is having confidence in our results, and confidence is generated by having a sufficient number of visitors that perform that KPI. This means getting sufficient traffic to make those decisions, and designing your experiments with that in mind. So before running a test, we get information from the customer about the traffic that they expect to reach the KPI, which allows us to design our experiments with the correct number of creatives based on running the experiment for some period of time - ideally for 2 weeks, but normally at least 1 week.

So testing revolves around the KPI, which means that if the KPI numbers are too low, you may NOT be able to:

  • Separate traffic into Personas,
  • Create a large template with a lot of variables and values - too many combinations to test

Since your low KPI will not allow enough time to reach the appropriate confidence level in a reasonable amount of time.

How does Optimost run these tests?

A generic definition is that Optimost uses the Internet to serve ASCII text using secure or non-secure connections. This is done using the web protocol HTTP or HTTPS. The content serve can include:

  1. HTML/CSS and/or javascript served to a web browser using a script include, and that can be:
    1. A Multivariable test (MVT) that test a number of page elements and their interactions.
    2. A/B/n tests that test different versions of a page
    3. Serving ads embedded in Web pages.
  2. Serving XML to:
    1. Flash embedded in a web browser
    2. A customer Server that uses the XML to generate a web page/email on the back end
  3. Hosting Landing pages for customers
  4. Serving ASCII text in some custom application

Most of our experiments – over 90% – are type 1.1 or 1.2. In these experiments, there is no interaction between the Optimost servers and the Customer's servers. The interaction is between the Optimost Servers and the visitor's Web browser, and once the browser loads the content, the changes to the page are made in the browser.

List of commonly-used Optimost terms

Optimost specific terminology

  • Technical Consultant (TC) - Optimost personnel who implement the technical aspects of an Optimost experiment.
  • Subject - Represents the individual Web page or content being tested. This can be a specific landing page, a test of the common header across an entire website, a test of Flash content, etc.
  • Trial - A trial is a test of a subject against a group of visitors (referred to as an audience). Initially the audience is tested as a whole. You can divide your audience into smaller personas if you want to optimize creatives for each particular persona.
  • Template - A template implements the testing portion of a subject and is the most important part of the optimization process for a TC. Templates contain all the content that will be optimized, and each area being optimized is a variable, each contain 2 or more values. Each subject can have one or more templates.
  • Creative - This is a single permutation - combination of variable and their values - for a specific template.
  • Control - The one creative that has the "control" values for each variable, basically the customer's original version of the page.
  • Challenger - A creative in a Wave that is being tested, has not been tested before, and is not the control.
  • Wave - A wave is a test of a specified set of creatives run against a persona of visitors.
  • Persona - A persona is a group of visitors with common traits that are participating in an experiment - for example, visitor who are Pay-per-click or not, IP addresses range, geographic location, some query string value in URL that tells us something about the visitor. Each Trial defaults to having one persona - the All Visitors persona - and other personas can be added when needed. TCs normally add a QA persona to all Trials for testing, and may additionally add Personas to group visitors as specified in the STP or to remove traffic from the test - ie problematic browsers or IP addresses internal to the customer.
  • Session Identifier - A unique value that associates the activity with a particular visitor and creative. It contains time/date information, along with creative, wave and visitor information. There is no Personally Identifiable information in this value.
  • Stickiness - The process of giving a repeat visitor the same experience when they return to an experiment. We normally drop a cookie in the visitors browser with the Session Identifier and if a visitor returns to the same wave in the same browser, and if stickiness is set, they will be served the same creative. Note that:
    1. The default setting for this cookie is 30 days, which is the maximum we would expect most waves to run.
    2. Stickiness is tied to a wave, so if you progress from Wave 1 to Wave 2, or switch into different Personas, stickiness does not apply and you will get a new creative.
  • Activity Tracking - Tracing the visitor's progress through the customer's funnel. We also use the Session Identifier as the value, but usually it has a different expiration date - normally 1 day - than the Stickiness cookie.
  • Set internal domains/Set clickthru domains - Setting that controls what target domains are allowed for clickthrus.
    • Set internal domains is a Portfolio-level setting for all clicks in all subjects in domain.
    • Set clickthru domains allows for more granular control for a given subject.
  • Refresh Creatives - A "snapshot" of a Template that is used to remake the creatives for an existing Wave, taken after the initial creation of that Wave. This is done to allow creatives to reflect the latest modifications to a Template(s) after Wave creation. Refresh is also needed for changes made to some Subject level settings, such as:
    • Activity Tracking
    • Stickiness
    • Changes to clickthru domains.
  • MVT - Multi-variable Test, an experiment that has more that one variable. Most of our experiments have 3 to 6 variables, and each variable normally has 2 to 6 values.
  • STP - Subject Test Plan, a description of the experiment page, the values being tested, and what will be done on that page.
  • Dynamic Content and Business Rules - The aspects of a web page that may change from view to view, and thus Optimost TCs must accomodate/account for when creating a Template.
    • Dynamic content - Content that changes for all users in all circumstances. For example:
      1. On a Product page, the image for the product, the price, the description can be expected to change for every product.
      2. On Forms, since the visitor may incorrectly fill in the fields, the form content has to account for the field values and error indication code.
    • Business rules - Content that changes based on the state of the page. For example:
      1. Logged in versus not logged in users may see different content on certain pages.
      2. On a Product page, some products may have selections/content others do not - for example a pair of jeans will have sizes, but a scarf will not.
  • Key Performance Indicator (KPI) - The success point for an experiment. Decided upon by customer and Optimost.


Web terminology relevant to Optimost

  • CSS - Cascading Style Sheets, used to create the formatting (how the elements are displayed) for an HTML document.
  • DOM - Document Object Model, a cross-browser representation of the nodes on a web page. Optimost TCs write javascript to modify the DOM and implement our changes (for the 90%+ of experiment in category 1.1 and 1.2 above).
  • First-party domain - The domain in the address bar. For Optimost, the domain is relevant to how we drop cookies to track activity and stickiness. Note that:
  1. There can be any number of subdomains within a domain. For example, acme.com is a domain, www.acme.com, shop.acme.com, and www.test.acme.com are different subdomains of that domain.
  2. For Optimost, we like to drop our activity tracking cookie in the base domain since the funnel may traverse different subdomians - and a cookie dropped in one subdomain cannot be read in different one - so we have written the javascript code that drops cookies to assume we have a 2-level base domain, find it, and drop the cookie there.
  3. If we are dealing with 3-level domains, which are normally international domains - acme.co.uk is 3-level, acme.uk is 2-level - we then need to hardcode the domain in the console. Both of these options are implemented in the Change Activity Tracking link at the Subject level, within the cookie tab.
  4. If you have an experiment that will be deployed over multiple funnels and at least one of them is a 3-level domain, you can write custom code to allow you to use first party tracking in those multiple domains. Here is a link to an example Dropping cookies in both 2-level and 3-level domains
  5. Since the Optimost code is returned from servers in the Optimost domain, cookies in the First-party domain can not be set by our servers, but instead can only be set using javascript in the browser.
  • Cross-domain or Third-party domain - When a web page contains script calls to domains that do not match the domain in the address bar, these are calls to Third-party domains. Note that:
  1. These cookies must be set by the Third-party server in the HTTP headers.
  2. These cookies cannot be read using javascript since javascript can only read cookies in domains that match what is in the address bar.
  3. For tests that have funnels spanning multiple domains, if we use cookies, we need to use cross domain cookies.
  4. Some browser now default to blocking Third-party cookies, so we must be careful when using them to track progress down the funnel.
  • First-party vs Third-party - shortcut to which is which - There are two ways to write/drop cookies:
  1. Cookies are set using javascript - then they MUST be first-party. Using javascript means you are doing it in the browser, and the browser only has access to its own domain (whatever is in the address bar).
  2. Cookies are set by the Server - if so, they can be EITHER, depending on what the domain of the web page is (whatever is in the address bar).
    1. If the base domain of the server and the base domain in the address bar MATCH , they are first-party cookies.
    2. If the base domain of the server and the base domain in the address bar DO NOT MATCH, they are cross-domain or third-party cookies.
  • First-party vs Third-party - shortcut to how/when you can read them
  1. Third-party - making a server call - Anytime you make any call to a server, all the cookies for that domain are returned in the HEADERS of the HTTP call
    1. That means ANY call, whether its an entire web page, or some content of that web page, like a javascript call, an image call, a css include, ANYTHING that calls a server.
    2. Cookies returned will be for your domain and the base domain only. So if a call is made to by.optimost.com, the optimost.com and by.optimost.com cookies are returned, but not asp.optimost.com or www.optimost.com cookies.
    3. These cookies are available to the Server, not in the browser (ie, in our Templates we cannot access these cookies)
    4. As people get more paranoid about the web, users are more likely to block third-party cookies out of "privacy concerns".
      1. Safari defaults now to blocking third-party cookies.
      2. We try to avoid using third-party tracking because of this.
  2. First-party - reading/writing cookies using javascript
    1. By definition, these will be in the first-party domain, and a subdomain/base domain when writting a cookie can be controlled by specifying it.
    2. We add functions to the Trial Code call to set the cookies
    3. The Page Code has a function to read cookies, and in our experiments we read first-party cookies that start with the letters "op" and send them back.
  3. See Page Code and Trial Code explained and Trial Code & Counter Code calls - details for more details.
