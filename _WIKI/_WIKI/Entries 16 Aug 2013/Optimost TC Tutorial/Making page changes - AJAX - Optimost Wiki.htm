INTRODUCTION

In a standard modular Optimost deployment, the content being tested is modified as the page loads. This is done by adding the Optimost code to the page so that it executes sequentially after the area being tested has been loaded – we normally place the Optimost “module” calls below the content to be tested.
However, if we are testing a page that contains AJAX, and the content and/or area being tested is modified using AJAX, then the customer’s AJAX code and the test area need to account for the presence of the Optimost experiment.

REQUIREMENTS

To support Optimization of a page containing AJAX, we need to determine:

  1. If the AJAX changes the test area in any way. This can include:
    • The AJAX call changing a test variable
    • The AJAX call changing the test area, but not variable.
    • Anything that results in changes made by Optimost being removed from the page
  2. If the AJAX changes do modify the test area, the client will need to implement a “hook” or call to Optimost immediately after the AJAX response from the Servers has been added to the DOM.
  3. If the client wants to account for the “flickering” as Optimost changes the content immediately after it has been modified by the client, then the test area needs to be made invisible while it is being modified by the client, and then to made visible again after the Optimost changes are implemented.

CLIENT IMPLEMENTATION ELEMENTS

The Client needs to make their AJAX call, and after the successful response is received and they make the necessary changes to the page, the client must then make a call to the Optimost code in the form of a function call.

In this example, the function reloadDiv(pUlr, pDivId, pCallback) contains

  • pUrl – the location of the client’s AJAX server, which is called in the function
  • pDivId – is the container that is modified by the AJAX call
  • pCallback – is a function that is called when the AJAX response is received. This may or may not exist.

What this function does is setup an AJAX call to pUrl, then waits for a response. When a response is received from the client’s servers, it adds the parsed response value as the innerHTML of the DOM object with id=pDivId. Finally, they can pass in a function using the pCallback parameter to be called after the changes are made.

"Timing of changes" - when the Optimost call is made

In most Optimost implementations, the call to the Optimost Servers is made as a standard Include, or during Page Load. For AJAX implementations, this is still normally the case, but there may be instances where calling the Optimost Servers is best deferred until after some "user action" on the test page - clicking some button, focusing on some field, etc. Since this means after Page Load, we need to be conscious of how the Browser's operation differs before and after Page Load.

Loading javascript content after Page Load

Most javascript functionality is identical before and after Page Load, with one major exception. The document.write command is used to write HTML to a web page, and is generally easier than adding nodes to the DOM as the page loads. For dynamically adding new HTML to a page as it loads, it works great. Think of a cursor working its way down the page as it loads, and a document.write will output contents based on where that cursor is. But when the page is finished loading, that cursor goes back to the top of the page. If you happen to use document.write at this point, you "write" the content to the top of the page, and in the process wipe out all the content on the page. Probably not what you wanted to do.

Optimost and after Page Load Trial Code calls

As mentioned earlier in this Tutorial, most of our experiments use modular implementations. They allow you to separate the loading of the Optimost code from the execution of that code by creating "modules" functions to be called later. As part of that, modular implementations have a built-in document.write for hiding the test area (to be later unhidden in the module) - and that cannot be removed. So for those instances where the Trial Code call is made after Page Load, we cannot have a modular implementation. Instead, we create an "Inline Web Page Snippet". When creating a Subject, must be define it as "Custom" in the Template Wizard and the "Type of subject" as an "Inline Web Page Snippet". In this case the content is all rendered inline - as in the Setup Block of a Modular Web Page - so if you need to defer execution, you must built any functions into the code (not a big deal, just that it will not be done automatically as it is for modular tests).
