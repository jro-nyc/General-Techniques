Client-side integration guide

For the Standard Modular Web Implementation - we add Page Code, Trial Code, and Module Code - we send the customer a Client-side integration (CSI) guide that explains how to add our code. It explains:

  1. That we work in the browsers, and that we need to add 3 pieces of code to their page as HTML in that page.
  2. This first is Page Code, which is a script block that goes normally at the bottom of the <head> section of the page.
  3. Next comes the Trial Code, which is another script block, and it goes just under the Page Code at the bottom of the <head> section of the page.
  4. Finally, Module Code is a script block that goes below the test area. If we have multiple modules, each goes below its specific test area.



As explained earlier:

  1. The Page Code is the same for all pages
  2. The Trial Code is similar from page to page but will have a unique URL for each Trial and may have unique Throttling parameters.
  3. Module Code is subject specific, but the names are often reused across pages.

So a CSI guide is sent out for every experiment/subject we create for the client, and the customer implements it for each experiment.

Practical Considerations - Number of Modules

  1. In most applications, Optimost TCs create a single module in a Template, and its usually named body. All the changes to the page are made in that module.
    • + Easier to manage code since its in at most 2 areas, module and setup block.
    • + Less changes to the customer's page in order to deploy the code.
    • – For pages that take a long time to load, the page will go from blank to suddenly visible with more delay than if we had multiple modules.
    • – You need to know the Optimost variable abstraction language to do this - all the TCs know this - which makes it harder for new customers to use as self-service.
  2. You can also create multiple modules in a template, and that is normally done with each module corresponding to a single variable.
    • All the + above become –, and vice-versa.
    • + The same paradigm that some other tools provide, so we can take over their existing test if we win their accounts.

TCs should continue creating as few modules as possible, unless there are page load issues that can be alleviated by going to multiple modules.

Practical Considerations - Quality of HTML of customer's page

Ideally, we would like the following to be true concerning all the test pages:

  1. The page HTML must be valid so that the DOM is parsed properly by all supported browsers.
  2. The content containers – the div, span, td, th, or other tags used to identify variables – must have unique IDs or class definitions that can be used to uniquely identify DOM nodes.

#1 is usually very important, although we unfortunately get some really "choice" HTML from certain customers. The issue is being able to make changes that are consistent across all browsers.
#2 allows us to not use <span> tags around the test area, which makes deploying our code much easier. We have had instances of the extra <span> tags breaking customer script functions that assumed the number of parent nodes between inner and outer node elements, so we try to avoid changing the page structure if at all possible.
I say ideally because we can sometimes work around certain HTML issues with strategic placement of our Trial Code and/or Module Code, but these page issue may increase the time-line or force us to segment out some browsers.

Practical Considerations - Tests spanning more than one page1

Occasionally, we will have a test across more than one page. They normally take one of the following forms:

  1. A single experiment over multiple pages. Which can be:
    • Multiple pages with a page load for each page and different modules used for each page. Views and visitor numbers will be skewed since there will be multiple views for a real visit that goes through all the pages in the test.
    • Multiple pages with a page load for the start page, and then the subsequent "pages" come in via AJAX as the previous page is "finished". This will also use different modules used for each "page".
  2. A Universal test that spans most or all of the website - say a test of the Header, Footer, or Left Nav areas that are similar on all the pages on the site. Again, the views and visitors numbers will be skewed.
  3. A common element test of something shown on a smaller subset of pages, such as an advertisement or disclaimer.

The most important thing to remember for multi-page experiments is avoid having modules using the same name. So whereas I recommend using body as the module for all your whole-page experiments with a single module, never use that for a multipage experiment. Make it something that makes sense given what is being tested - say "header", "left nav", "advertisement 1" for type #2 or #3, and "page 1", "page 2", or "persona info page", "credit card page" for type #1.

1See Multi-page experiments for more information

Practical Considerations - Pages with multiple tests

So when you have a Universal test running on a website - say the header - you are almost guaranteed to have multiple test running on some subset of the site's pages. Say you have a Universal Header test and you are also running a test on a Product Page.

  1. You need to make sure the the modules used do not have the same name, as explained above.
  2. If there are two experiments going on, then there are two trial code calls going on. If, to set stickiness, you set the sticky=123 parameter to pin a creative, know that if both subjects (experiments) use optrial.sticky under "Force sticky request argument:" in the Change stickiness popup, at least one of them will show failover copy2 since they cant both have the same creative running. You must use the debug tools we have for these instances. See Optimost Debugging for more information.

2For more on failover copy, see Failover copy or Optimost Creative Serving network
