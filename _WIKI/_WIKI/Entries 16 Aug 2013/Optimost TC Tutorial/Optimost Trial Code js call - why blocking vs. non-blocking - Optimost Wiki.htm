Blocking vs non-blocking (Asynchronous) javascript calls

As we could see from the entry Page Code and Trial Code explained, our Trial Code call adds the script tag using a document.write. This creates what is called a blocking javascript call - all the content has to load AND execute before the web page can move its cursor to the next item on the page. For javascript that does not have to be loaded and executed before any other code on the page, there are techniques that allow for "parallel" execution of javascript - non-blocking - resulting in faster page load times and thus improved performance.

Non-blocking techniques (Asynchronous)

There are a number of ways to do this, but in our scenario - Optimost as third-party content on a customer page - we only need to concern ourselves with:

  • Adding the Trial Code call as a script element added to the DOM

In the entry Page Code and Trial Code explained, at the bottom, the section Modifications: How to perform Trial Code call AFTER Page Load explains a way to do this - replace optimost.R() with code adds the tag to the DOM.

Problems with non-blocking

So why does Optimost use blocking code? As you know our code consists of Page Code, Trial Code, and then Module code further down the page that calls a function defined in the content returned by the Trial Code call. When testing non-blocking code, we found:

  1. In some browsers (in the past -before 2012 - it was only in IE), we often had the content of the Trial Code call returned AFTER the module call was called
  2. This means that the module was NOT executed and our code did not run
  3. And the page appeared blank, the worst of both worlds.

If the customer insists on having non-blocking code...

I mention this because we have had customers ask us why we don't use non-blocking code. Its all the latest "craze" and not doing it says we are out of touch. Here are the pertinent issues:

  1. If hiding the content and then unhiding it after we make our changes is important - so that you don't see the control followed by the "flicker" of the content changing to the challenger - then using a non-blocking code will not work. You may see the "flicker", depending on when the Trial Code call is executed.
  2. If they are OK with the flickering, then we would need to modify our code to account for the possibility that the Trial Code call is returned AFTER the module call. So you will need to:
    • Add an additional module call as the last line in the setup block (adding it to the template) with a conditional execution statement.
    • Modify the normal module call to first define some global variable, then make the module call.
    • Inside the module, look to see if the variable is defined. If it is execute the code, and set that variable to null. If it is not set, exit out of the function.
  3. For the TC, if you are doing an A/B test and they are using non-blocking code, you have to add elements to the DOM, so you cannot have pure HTML in your template.
  4. Modify optimost.R() as specified in Page Code and Trial Code explained to add the script element tag to the DOM.
