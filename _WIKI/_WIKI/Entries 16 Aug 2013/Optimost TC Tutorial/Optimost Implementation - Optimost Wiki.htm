How does a customer implement the Optimost solution?

In order to run an experiment, the customer must set up the "end device of interest" to call the Optimost servers, which then returns the necessary content to implement the test. The end device is usually:

  1. The web browser of a visitor coming to the customer's site (>95% of applications).
  2. A customer's server - web, email, or some other application (<5% of applications).

Web browser-based implementations

The vast majority of our implementations are in a web browser, over 95% of the tests we run. They can be broken into the following subcategories

  1. A "standard" implementation where some part or all of a web page is tested
    • The call to our servers is made from the web browser via a javascript include
    • The customer's page that is being tested is specified before the test, and an STP tailored to that page is created
    • The changes to the page are implemented by Optimost Technical Consultants with javacript, CSS, and HTML.
  2. An "XML Flash" implementation where we only test a Flash object(s)
    • The call to our servers is made from the Flash Object as an XML call
    • The Flash designers create Action script that calls the Optimost servers, then takes the Optimost XML data response and displays the appropriate content
  3. An "Ad" implementation where an ad of some form factor is tested on different web pages
    • The call to our servers is made from the ad entity
      • It can be a standard javascript include
      • It can also be an XML Flash implementation
    • As with most ads, the page where the ad will be placed is undefined and the experiment is focused on the ad, not the page containing the ad (although implementation issues that need to be considered will be accounted for)

Standard Implementation

Standard Implementation usually take the least amount of work on the part of the customer, and they consist of over 90% of the browser-based implementations. The Standard Implementation requires the customer to add 3 pieces of code to their page.



  1. Page Code - An Object that creates functions/methods to pull relevant data from the web browser to send the Optimost Servers.
    • Exactly the same for every experiment
  2. Trial Code - Calls several Page Code functions to construct the URL used to call the Optimost Servers, and makes that call
    • Very similar for each experiment, but:
      • Will have a unique URL (different for each experiment)
      • May have unique "throttling" parameters
    • The code make a script call to our servers, and our servers return, in order:
      • Code to write first-party cookies - for stickiness and activity tracking - if one or both are set as first-party cookies.
      • The code for the defined modules, which are functions to be executed later.
      • The setup block, which is executed inline.
      • You can see Trial Code & Counter Code calls - details for more details.
  3. Module Code - One or more modules calls placed just below the corresponding test area(s), they execute the module functions returned in the Trial Code call.

For more information on the Page Code and Trial Code, go to:
Page Code and Trial Code explained

Global Implementation

A Global Implementation is a Standard Implementation for customers that want to deploy the code only once across their entire site.



  1. Page Code - An Object that creates functions/methods to pull relevant data from the web browser to send the Optimost Servers.
    • Exactly the same for every experiment
  2. Global Code - This replaces Trial Code in this implementation. The Global Code consists of:
    • Selector Code - The code that physically replaces the Trial Code on the customer's page. Allows us to select between a "staging" and "live" version of the Global Code for testing purposes.
    • Decision Logic - The logic returned by the staging or live Global Code, this logic determines if the webpage has an experiment on it or not. If so, then a Trail Code call is made for the appropriate experiment, as determined by the logic.
    • Trial Code (IF SELECTED) - Same as the Standard implementation, it calls several Page Code functions to construct the URL used to call the Optimost Servers.
  3. Module Code - A module calls placed at the bottom of the page so it will cover all potential test area(s), they execute the module(s) returned in the Trial Code call.

Customer Server-based or Server to Server implementations

Comprising less that 5% of our implementations, customers employ this method usually for one of two scenarios.

  1. A web page that is deemed too sensitive to have a third-party javascript include that manipulates the content on the page.
  2. They want to run an Email test

Some end device that want ASCII text in some custom form

Although at present most devices use defined standards for the ASCII text they receive - XML, HTML, etc - if there is an end-device that needs some custom implementation, Optimost can also be used to interface with it and conduct a multi-variable test. We would, of course, need the specifications for the data that is expected, but we have the facility to implement such an experiment.
