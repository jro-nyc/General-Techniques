Here we will explain how the Page Code and Trial code work together to construct the URL to call the Optimost servers and pass all the necessary parameters in that call. For Optimost TCs, knowing how this code works is an essential part of what it means to be a TC. Answering Customer's question on how Optimost works and what Optimost can do without knowing how the Page Code and Trial Code work comprehensively is almost impossible.

<script language="javascript" type="text/javascript"><!--
var optimost={
	A:{},
	C:{},
	D:document,
	L:document.location,
	M:[],
	Q:{},
	T:new Date(),
	U:'',
	V:'2.7',
	Enabled:true,
	ST:"script",
	SA:{"type":"text/javascript"},
	//I - gets the cookies and query string parameters.  Notice that this is called at the bottom here, above the trial code.
	I:function(){
		var s=this.L.search;
		var c=this.D.cookie;
		if(s.length>3){
			for(var a=s.substring(1).split("&"),i=0,l=a.length;i<l;i++){
				var p=a[i].indexOf("=");
				if(p>0)this.Q[a[i].substring(0,p)]=unescape(a[i].substring(p+1));
			}
		}
		if(c.length>3){
			for(var a=c.split(";"),i=0,b=a.length;i<b;i++){
				var v=a[i].split("=");
				while(v[0].substring(0,1)==" ")v[0]=v[0].substring(1,v[0].length);
				if(v.length==2)this.C[v[0]]=unescape(v[1]);
			}
		}
	},
	//B - puts together all the parameters that will be appended to the Trial Code URL.  This includes looking for optrial and adding its contents
	B:function(){
		var n;
		this.A={};
		var _o=this;
		this.A.D_ts=Math.round(_o.T.getTime()/1000);
		this.A.D_tzo=_o.T.getTimezoneOffset();
		this.A.D_loc=_o.L.protocol+"//"+_o.L.hostname+_o.L.pathname;
		this.A.D_ckl=_o.D.cookie.length;
		this.A.D_ref=_o.D.referrer;

											// are added to our array of stuff to be sent back on the trial code call

		for(n in this.C)if(n.substring(0,2)=="op")this.A[n]=this.C[n];
	},
	//S -  Construct the full URL, makes sure all the parameters for the QA are valid, and then appends it to the Trial Code URL
	S:function(){
		var q='';
		for(var n in this.A)if(this.A[n]!=null && this.A[n]!="")q+=(q.length>0?"&":(this.U.indexOf("?")>0?"&":"?"))+n+"="+escape(this.A[n]);
		return this.U+q;
	},
	//SC - Cookie setting code, like opCreativeSetCookieA() in the Trial code content
	SC:function(n,v,e,d){
		var de=new Date();
		de.setTime(de.getTime()+e * 1000);
		this.D.cookie=n+"="+escape(v)+((e==null)?"":("; expires="+de.toGMTString()))+"; path=/"+((d==null)?"":(";domain="+d));
	},
	//SLD - Domain finding code, like like opCreativeGetDocumentSLD() in the Trial code content
	SLD:function(){
		var sld=this.D.domain;
		var dp=sld.split(".");
		var l=dp.length;if(l<2)sld=null;
		else if(!isNaN(dp[l-1])&&!isNaN(dp[l-2]))sld=null;
		else sld="."+dp[l-2]+"."+dp[l-1];return sld;
	},
	//R - Traffic throttle code
	R:function(r,c,d,e){
		if(this.Enabled){
			var b=true;
			if(r<1000){
				b=(Math.floor(Math.random()*1000)<r);
				if(c!=null){
					if(this.C[c]!=null)b=(this.C[c]!="mvt-no");
					else this.SC(c,b?"mvt-yes":"mvt-no",e,d);
				}
			}
			if(b){
				var t='<'+this.ST+' src="'+this.S()+'"';
				for(n in this.SA)t+=(" "+n+'="'+this.SA[n]+'"');t+='><\/'+this.ST+'>';
				this.D.write(t);
			}
		}
	},
	
	addModule:function(s,f){
		this.M[s]=f;
	},
	
	displayModule:function(s){
		if(typeof this.M[s]=="function")this.M[s]();
	},
	
	hasModules:function(){
		return count(this.M)>0;
	}
};
optimost.I();  //This is executed right after the optimost Object definition, and is part of the Page code.
//-->
</script>

Details on what happens

Lets run through what happens with the page code and trial code, using the code that is defined above as a working example.

  1. First, the Page Code:
    1. The optimost object is created. It is a container with methods and properties that are needed for Optimost experiments.
    2. Once its created, the object's optimost.I() method is called. This method pulls out all the first-party cookies and all the query string parameters on the test page. The cookies are saved in optimost.C and optimost.Q holds the query string parameters.
  2. The optrial object, if needed:
    1. Next in the HTML example above comes a definition of the optrial object and property definitions for it, which is used to pass parameters back to the Optimost Servers through the optimost.B method. optrial is not always there, but if it is, I will explain later what happens.
  3. Then the Trial Code:
    1. The trial code gets "executed" - meaning the URL for this trial gets "relevant data" appended to it as query string parameters, and this string is used as the src value for creating a script tag in the HTML of the test page, which will then call the Optimost servers. The URL is defined in optimost.U, and later optimost.B() is called to place all the relevant data in optimost.A, which consist of the following information:
      • A time stamp
      • The time zone
      • The document location
      • The length of the document.cookie object
      • The document referrer
      • Any properties of the optrial object - if it is set by the customer above the trial code.
      • All query string paramaters - optimost.Q.
      • Cookies in the first-party domain that start with “op” - filtering optimost.C.
    2. Next, if we had defined page view attributes to be passed in, there will be code here to pull those out. In this page, we are looking for a javascript variable called is_sticky.
    3. Finally, optimost.R is called to make the call to the Optimost servers. This function works as follows:
      • 4 parameters are passed in, which are:
        • r which is the "throttling" percentage. Defaults to 1000 for serving to all visitors, 0 is for none of the traffic
        • c which is the name of the cookie that is used to track the "throttle state" for return visitors. Defaults to null for r=1000 throttling, otherwise the gum cookie name should be here under normal circumstances (more on that later)
        • d which is the domain for the cookie, again defaults to null for r=1000, otherwise optimost.SLD() gets the domain, unless its a 3-level domain, then you need a string or a new function for it.
        • e which is the time duration for the cookie in seconds, again defaults to null for r=1000, otherwise usually 2592000 which is 30 days in seconds.
      • If it is deemed that we are in the experiment, then a tag is created based on the contents of optimost.U, optimost.A, optimost.SA, optimost.ST, and optimost.S is called to construct the URL.

More on optimost.R(r,c,d,e) and Throttling

Since Throttling has to be well thought-out when implemented, here is more detail on how optimost.R() works and some implications of its functionality:

  1. First, we look to see if optimost.Enabled=true, which it does by default. If it is false, we drop out and no trial code call is made.1
  2. Otherwise, we begin our "call or not call Trial Code" decision by setting b=true.
  3. If r>=1000, we skip the next if and, since b=true, make the Trial Code call.
  4. If r<1000, we then generate a random number between 0 and 1000, and if the number is less than r, b=true. If not, b=false.
  5. If c!=null - c is a defined parameter (not null) in the optimost.R call - then we look to see if a cookie with the name of c exists - checking optimost.C[c] for it.
    1. If the cookie exists, then if its value is "mvt-no", we set b=false. If it has any other value, then b=true
    2. If the cookie does not exist, we set the cookie to:
      1. mvt-no if b=false
      2. mvt-yes if b=true
  6. We then fall out of the if and if b=true, we make the Trial Code call.
    • The call is a performed by creating a string with the following content:
      • A tag of type optimost.ST - script in this case.
      • Next, we call optiomst.S which creates the URL by combining:
        • optimost.U - the URL for this Trial.
        • Then taking each property of optimost.A and turning them into name-value query string paramters for that URL.
        • And the parameters in optimost.SA are also used to create the tag.
    • This string is then put inside a document.write and rendered on the webpage (wherever the cursor happens to be, that's how document.write works).
    • And since we use document.write, this means a Trial Code call based on optimost.R has to occur BEFORE page load. To do Trial Code calls, you need to make a new version of optimost.R, see Modifications: area below for more details.

1Setting optimost.Enabled=false; is a quick and easy way to turn off the Optimost test without having to remove the code.

Throttling - more details

A few things to think about when throttling:

  1. When we throttle, we normally want to make a throttling decision the first time a visitor comes to the page, then continue with that decision the next time they return so they will have a consistent experience. So remember that if you have Trial code with _o.R(1000,null,null,null) and you want to add throttling, you need to fill in the other 3 paramters with the cookie name, domain, and time. If you leave them as null, you will "roll the dice" every time the visitor returns to the page, which is normally not what you want.
  2. If we have throttling to some non-100% value, and we want to change it to some new, non-100% value, understand that return visitors will continue to get their old behavior if you do not change the cookie name.
  3. If we have throttling to some non-100% value, and we want to change it to 0%, you can:
    1. Set optimost.Enable=false to do this
    2. Set the cookie value to null so that it is ignored, and the random number will never be 0, no one will get into the test.
    3. Set the cookie value to a new name if you want some way of later knowing who came to the page and was throttle out.

The randomness of Math.round()

if you look at the function optimost.R, we do throttling using the javascript function Math.random(). The best way to generate random numbers is through physical means - flipping coins, rolling dice, a lottery ball drawing - and it is difficult for computer functions to generate true randomness because the initial value chosen - the key - may not generate a truly random distribution of number. You may read articles saying this about Math.round(). However, this lack of true randomness results in specific values coming up more than they should, NOT in an uneven distribution of values. For Throttling, we only care about the distribution of values, so the randomness of Math.round() is more than sufficient.

Non-blocking or Asynchronous calls: How to perform Trial Code call AFTER Page Load (Also applies to Counter Code)

The last function call in the Trial Code is a call to optimost.R, which makes the throttling decision and adds a <script> tag to the page by using a document.write to render it, and that tag calls the Optimost servers. This document.write must happen before Page Load. If, for some reason, you need to do a Trail Code call after Page Load, you need to replace the document.write of the <script> tag with code to add a script element directly to the DOM. To do this, we modify the code as follows:

Changes to Page Code

You need to add the following: 

optimost.isIE6=(navigator.userAgent.toLowerCase().indexOf("msie 6")!=-1?true:false);
optimost.XH=function(u){
    if(typeof(u)!="object")return;
    var s=this.D.createElement(this.ST);
    for(var n in this.SA){
	    if(!this.isIE6)s.setAttribute(n,this.SA[n]);
	    else s[n]=this.SA[n];
    }
    for(var n in u){
	    if(!this.isIE6)s.setAttribute(n,u[n]);
	    else s[n]=u[n];
    }
    var h=this.D.getElementsByTagName("head");
    if(h[0])h[0].insertBefore(s,h[0].childNodes[h[0].childNodes.length-1]);
    else this.D.body.insertBefore(s,this.D.body.childNodes[this.D.body.childNodes.length-1]);
}
optimost.RXH=function(r,c,d,e){
    if(this.Enabled){
	var b=true;
	if(r<1000){
	    b=(Math.floor(Math.random()*1000)<r);
	    if(c!=null){
		if(this.C[c]!=null)b=(this.C[c]!="mvt-no");
		else this.SC(c,b?"mvt-yes":"mvt-no",e,d);	
	    }
	}
	if(b){
	    var u={"src":this.S()};
	    this.XH(u);
	}
    }
}
  1. optimost.isIE6 - Need to determine if the browser is IE6 because of how a script element for IE6 musts be constructed.
  2. optimost.XH - The code to add the new tag as an element to the DOM.
  3. optimost.RXH - The modified version of optimost.R that calls optimost.XH instead of constructing a script tag and using a document.write to render it.

These can be added right after the Page Code is defined. Notice that I did not redefine optimost.R - which I could have done by adding an extra input parameter and still make it backwards compatible - but instead created a new function optimost.RXH so to make it clear/obvious to someone looking at the code that we are doing a custom modification here.

Changes to Trial Code

Using the no throttling example from above, the new Trial Code needs to look like the following: 

<!-- OPTIMOST TRIAL CODE V2.6 - Copyright 2002-2011 Optimost LLC -->
<script language="javascript" type="text/javascript"><!--
(function(){
	var _o=optimost;
	_o.U="http://by.optimost.com/trial/2/p/isoptrialvisibleintemplate.efe/94/content.js";
	_o.ST="script";
	_o.SA={"type":"text/javascript"};
	_o.B();
	_o.A["is_sticky"]=(typeof is_sticky!="undefined")?sticky:null;
	_o.RXH(1000,null,null,null);})();
//--></script>
<!-- END OPTIMOST TRIAL CODE -->

This is the "no traffic throttling" example. For "traffic throttling" example, it's the same, replace _o.R with _o.RXH

Changes to Template Content

Since we cannot have document.write in our code, make sure there are no style blocks on inline HTML in your templates. You need to use the createStyle function (Create Style) to add styles and add any inline HTML to a string and add it as the innerHTML of a new node, then add it to the DOM. And for script or link tags, add them to the DOM also.

Modular Web Page Snippet templates: Changes to Modules setting

The contents coming back in the content.js call from the Trial Code call must also NOT contain a document.write. This is an issue because modular web page snippets subjects default to having an automatically-generated style definition that is rendered using a document.write. This style definition hides the "span tag" that used to be wrapped around the test area. Although we rarely use the span tags, the system still defaults to having this feature. As a workaround, you can go to the "Define Modules" at ths Subject level and set the Integration style to "After Document Load".

Inline Web Page Snippet: Subject that will accommodate Trial Code call after Page Load out of the box

Although rarely used these days, know that these are ready out of the box to be loaded after page load. Not recommended to be used normally, but know that for after page load applications, may be acceptable
