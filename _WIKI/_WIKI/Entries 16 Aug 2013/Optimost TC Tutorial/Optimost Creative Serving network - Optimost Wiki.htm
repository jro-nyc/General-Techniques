High-level Architecture for Creative Network

Here is some diagrams of how the Creative Serving Network is configured, with explanations.

Standard approach: Browser-based experiments

The first diagram show our standard implementation, where Optimost code is place in the customer web pages to be tested.
A few things to note:

  1. As we remind customers, in a standard implementation, there is no communications between the Optimost Servers and the Customer's Servers. Everything happens in the Browser.
  2. These experiments usually require relatively little work from the customers in creating experiments since Optimost Technical Consultants or the Author tool will create code to make the necessary changes to the page being tested.
  3. The final product served by Optimost is javascript that changes the DOM or CSS of the page.
  4. The overwhelming majority of our experiments - 99%+ - are standard implementations.

Specialized approach: Server to Server experiments

This second diagram show a Server to Server implementation, where Optimost code communicates with the customer's Servers, and then the Customer creates the experiment based on this content.

A few things to note:

  1. Unlike the standard approach, the communications is between the Optimost Servers and the Customer's Servers. Optimost does not have access to the final product, so the Customer will be responsible for creating that experience based on the content received from Optimost.
  2. These experiments usually require more work from the customers in creating experiments since they have to create the final user experience. Normally, Server to Server experiments are either:
    • Email experiments, where you cannot use a javascript to modify content (as a Standard implementation can)
    • Any browser-based experience that is critical enough for security reasons that the customer does not want javascript code manipulating the page.
  3. The final product served by Optimost is XML that encapsulates what the customer has asked for. Optimost works with the customer to create the appropriate XML schema for transporting the desired content.
  4. Although comprising less than 1% of our experiments, Server to Server can be very attractive to large customers with the resources and volume to profit from multivariable testing.

For all Optimost experiments

The following apply to the serving network, and are the same for all experiments

  1. Akamai routes the traffic to the closest/fastest responding Optimost Server, one of the 5 location worldwide.
    • The by.optimost.com/by.essl.optiomst.com domain points to Akamai, and then Akamai figures out where to route the request.
    • Akamai does not host the Creative Servers.
    • We do have content hosted by Akamai, our Uploads on the es.optimost.com domain.
  2. If our Creative Servers are down/not responding, Akamai has an offering that takes care of that scenario, as we shall see next.

Storing and Presenting the Analytics data

Here is a detailed diagram of our Creative Serving Network and the Storing and Presentation of the data.

When any visitor event occurs - a page view, a clickthru, a counter being triggered - that event is routed through Akamai to our Creative Servers, or BY Servers. That data is stored at the BY Server's DB, then pushed to our Pleasanton Data Center every 5 minutes. This data then goes through the "sharding" process, controlled by App-ple-01, being transferred to one of 5 DBs, or shards. In the diagram, Db-ple-01 to Db-ple-05.

For redundancy, each shard has a backup, and that data is backed up at the same time as it is sharded, in the diagram to, Db-ple-06 to Db-ple-10. We then back up the entire DB from Db-ple-06 to Db-ple-10 once a day, to Backup-ple-01. In addition, we also have a 2-week buffer that is stored in App-ple-03 for storing the raw data BEFORE the sharding process so, if need be, we can reconstructed the last 2 weeks of data at another location.

When accessing the Agency Console, it will get the Analytics data from Db-ple-01 to Db-ple-05.

Akamai and Uptime protection

When Optimost was formed, the first question from every customer's tech department was "what kind of uptime guarantees can you give us". We could talk about how great our network was, but all they would see is a couple of guys in a startup saying to a multi-million dollar business: "trust me". Fortunately for us, we were able to assuage their fears by telling them we used Akamai Site Failover to Edge Server to serve our content in the off chance that our servers are down.

Failover Copy

Failover is done by allowing Akamai Edge Servers to route all visitor requests to the Optiomst Creative servers through their network. And if they can not get to our Creative servers, they would instead route the traffic to a "failover" copy hosted by the Akamai Edge Servers that contains the content for the specific subject that was being requested.


In our early experiments, we did HTML replacement of content, not DOM modifications, so the customer wrapped the HTML we where testing in a <noscript> block and we added our old-style Trial Code call below that (this is before Page Code). If our servers where non-responsive and we had no Failover copy, the visitor to the test page would see a blank test area. So we would put the default version of the test area in failover, which was then and is now at the Template subject level under the left nav entry "Change failover copy".

With how we implement most of our tests today - modify content via the DOM so that if we do nothing, the default page show up as normal - we simple leave failover copy empty, but if you do have a test where sending no content would be a problem, you need to use failover copy. For example:

  1. Any XML test, which the end device will always expect an XML response with valid data.
  2. Any redirect test where the control creative is - and usually ALL the creatives are - a redirect. You will run into customers that want to use Optimost to manage their content like this, so if they want this, you will need to add failover to redirect to the control page.
  3. A test where we are changing the default version of the page and the Customer never wants the existing default version - the one in the HTML - to appear.

For more on failover copy, see Failover copy

Optimost Creative Servers - NOT CACHED

For an experiment, Creatives are served to different visitor Personas. Creatives are not cached because each creative contains unique tracking information - a session identifier - that is generated each time a creative is served. So each time a visitor loads a page that contains an Optimost experiment, if a Trial Code call is generated (assuming throttling has not turned our experiment off), Akamai will route that request to the nearest Optimost Creative server, and the Optimost Servers will return that creative. Since our Servers have to generate each creative for each request, this tends to be an "expensive" in terms of our Server resources.

Optimost "uploads" - CACHED CONTENT

We also have the capability to host content that Akamai caches on their Edge Servers. This facility is accessible at the Subject level via the "uploads" link in the left nav. This is where we should be hosting our global includes - see Options - how to add Optimost code to Customer pages - and MAYBE content that the customer cannot host - images, JS includes, or other content1. Caching the content is "cheap" in terms of our Server resources since our Serves are not called, but it still has a cost associated with it based on traffic served.

There is a downside to cached content: the first time you upload it, you should see the content immediately. But when you change the content, it may take a bit of time for the changes to ripple out to all the cached copies across the Akamai network. So you may have people (or you) complain that they are seeing the old version of an uploaded entity, not the new one. The caching process is not perfect - you want perfect, then don't cache it - but it is much more economical. 1Know that this content costs us money to serve, so if you are hosting content for them, you must make sure it is allowed contractually.

Changes to Served content

Within Optimost, there are a number of different scenarios under which a TC or OM may want to change the content that is being served by our Creative Servers.

  1. Creating a new Wave for testing or QA
  2. Creating a new Persona for testing or QA, then creating a new Wave inside of it.
  3. Taking an existing Wave, making changes to the underlying template or templates, and then "pushing" those changes to the Creative Servers.

Creating a new Wave

When you create a new Wave and are finished adding creatives, you will then have to "activate" the wave before being able to push the content to the Creative Servers. This is a very important process. When you "activate" the wave, a "snapshot" of the template or templates used in the Wave is taken and is used for these creatives. So if you make changes to the underlying template(s), they will NOT be reflected in creatives unless you re-activate the Wave. The process of reactivation is called Refreshing Creatives.

Refreshing Creatives

Refreshing creatives is the process of taking that template snapshot again. If you are a TC doing QA on a template, you will become accustom to doing this over and over again on existing Waves. Here are a few things to remember about this process.

  1. When you make changes to a Template, you can refresh at the Template OR Wave level - This is very important when you have a template being served to real customers and you need to test/make changes without effecting the test, you can also have a QA Wave running and only Refresh Creatives on the QA Wave until you are ready to push the changes to the real customers.
  2. If you make other changes to the Subject that effect the test - say you change the Stickiness or Activity Tracking values - you will also have to Refresh Creatives - Stickiness and Activity Tracking are also passed in the creative, so you need to refresh to change these, too.
  3. Persona changes on an existing Persona DO NOT require you to Refresh Creatives

For more on Refresh scenarios, see the bottom of Testing and the Agency Console
