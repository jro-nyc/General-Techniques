As one would expect, experiments are usually run on a single page. We think of the "Home Page" or the "Product Page" as likely targets for optimization. However, there may be instances where we want to run an experiment over multiple pages as one single experiment - one subject. Here we will cover the different scenarios that can be implemented as a multi-page experiment.

The site-wide test

The most common multi-page experiments we run are on elements that exist on all or most of the pages on a website. For example:

  1. Headers
  2. Navigation bars
  3. Footers

These are elements that a customer may want to test to see if a different "look" or "feel" to the website results in better visitor response. The experiment is implemented much like a normal experiment, except that:

  1. The Trial Code call is made on every page on the site
    • So for a Global implementation, the Trial Code call is executed on every page.
    • For a standard CSI implementation, the Page Code/Trial Code and Module Code is inserted on every page
  2. The Module Code should be placed directly under the test area's HTML, as opposed to just at the bottom of the page as is usually done for a single-page experiment.
  3. The Module Code call should have a name that will not be reused in another subject/experiment, ie for single page experiments, we can generically call the module "body" across a bunch of different subjects, here we can use "header", "footer", etc, and make sure you never reuse those names in any other subjects.
  4. Views will be multiples of visitors since there will be multiple page loads as a visitor progresses down the funnel, so those numbers will not look the same as a single-page experiment. You will need to account for the difference.

The multi-page element test

This would be an element(s) that exists on a subset of pages across the website, for example:

  1. An on-site advertisement
  2. Some element(s) that appear on a type page containing related content, such as:
    • A security logo or logos on checkout pages.
    • A common testimonial on product pages.

The implementation considerations are the same as the site-wide test, except that the Trial Code call is now made on this common subset of pages only.

Hybrid multi-page - one module site wide, other module page-specific

This would be a test of a page on the site that you get to by performing some action elsewhere on the site. The action you take on the previous page determines the display format of the test page.

For example, we want to test a Search Results page. On the site, every page has a search tool in the header, and when you land on the Result page, it defaults to showing results by relevance. But once on the Results page, you can select to view result by:

  • By Price
  • Alphabetical
  • Highest Rated

And also change the view to be in a style, either:

  • List
  • Grid

How do we test this page so we can test the display format of the results and the buttons, position of price, image, fonts, on the results page?

Implementation

So we would have a site-wide module called "Search form", which is placed under the header on every page with search. In that module, we control the inputs passed in the search to dictate the display format on the Results Page. Then on the Results Page, we also place a module at the Bottom named "Results page" that makes changes to the Results Page elements only.

The multi-page funnel/flow test

In this scenario, we are testing two or more pages that we consider to be a single test. They may or may not have common elements, but we choose to test it as a single experiment/subject. Reasons for doing this include:

  1. You want to ensure that improvements on one page do not negatively influence other pages, and if so, detect this within the experimental Waves and figure out what cross-page combinations work best.
  2. You want to test the number of pages in a funnel, ie does having more or fewer pages in the funnel result in increased performance.
  3. Your "multi-page funnel" is actually a single page load followed by AJAX calls for the "next" page as you "finish" each step, so it's easiest to test as one experiment/subject.

The funnel/flow multi-page experiments may sound like the best way to go in a lot of scenarios, but understand that there are a number of drawbacks to this approach, including:

  1. You are adding a lot more variables, values and complexity to a template. This means:
    • You will need more traffic to reach statistical significance for your Waves
    • You are adding complexity to the template, and usually with no economy of scale. It will usually be harder to do the one 2-page template than two 1-page templates.
    • Using the Optimost Design Generator to generate a design automatically may not be possible if the variable/value numbers are too high.
    • QA time also increases to usually more than it would take to test the different pages individually.
  2. Changing the number of pages in the funnel usually requires involvement from the back-end development team, which may not be feasible because:
    • Resource issues - the dev team may not have a resource available to work on this.
    • Time issues - it may delay the experiment too long to get the dev team involved.
  3. When looking at the Analytics information, you will need to account for the nature of the experiment in mind.
    • Views and Visitors won't mean what they usually do - Views will be multiples of the Visitors in the analytics, depending on how many pages there are in the funnel.
    • TCs will have to add counters to the different "pages" to determine the real Views for each page.

It's important to know that multi-experiments, although perfect for some scenarios, are not the norm, and you need to make sure you have a good reason to implement one.

Implementating a funnel/flow test

For a funnel test, the implementation goes as follows:

  1. The Trial Code call is made on every page in the flow.
  2. There will be separate modules in the Template for each funnel page, and you will put only that page-specific Module Code on the appropriate page.
    • You can give the Modules numeric designations, ie you could call the modules "page1", "page2", etc
    • You can also name the Module based on function - "personal info", "credit card info", etc
