We will cover setting up experiments in the Agency Console and things you need to consider/look for when setting them up.

Agency Structure

First, let's look at the structure of the Console from a Hierarchical perspective.

Top to Bottom

The diagram below shows the complete Hierarchy of the Console.

  1. Agency - The Agency level is where you (the TCs) normally login. This contains all Customer Accounts serviced by Optimost. When Optimost was formed, there was only one Agency - the Optimost Agency, the one you will be using normally - but since we developed a self-service offering, we have also created other Agency Consoles for our larger Digital Agency partners, such as Digital River. Optimost/Autonomomy user credentials are added and controlled at this level.
  2. Account - Next come the Accounts, which are our Customers or any Optimost-created test environments. You, the TC, will be assigned specific Accounts to service. For Customer access to their own Account, their user credentials are controlled at this level.
  3. Portfolio - Below the Account level,you can associate your experiments into Portfolios based on some common association that you, the Account manager, deem relevant. This can be:
    • A single Portfolio for all our experiments - Some customers see no need for further separation of their experiments
    • A Portfolio for every brand - Some customers have different websites/brands they are testing - whether they are Digital Agencies or single companies with multiple websites/brands - and they use Portfolios to group their experiments.
    • A Portfolio for different types of test - Some customers like to separate Landing Page test from Product Page tests
  4. Subject - The "Page" or "Common Area" you are testing. At this level and below is where a TC does the majority of their work. From an Optimost Perspective, a subject can be:
    • A standard single-page experiment (about 90%+ of our experiments)
    • A site-wide element test, like a header, footer, left navigation, etc
    • An element test over a subset of pages - say testing a common upsell on Product Pages.
    • A multi-page funnel/flow test, a test spanning multiple pages that you deem must be run as one experiment.
    • Go to Multi-page experiments for more information


  5. Template - The Actual Code the TC adds to create the different variables and values for the experiment. For the TC, this is the most important part of the Optimization process.
    • Normally, a subject has two templates in it:
      • A Template with all the variables and values, usually named ver 1.0 (or version, v, no matter). This template will be used for all the waves in the experiment, normally by doing DOM modification.
      • A second blank template for showing the customer's default content on the test page.
      • Over time, the TC may create modified version of the ver 1.0 template for any number of reasons - customer changes to the STP or changes to their website, and may use a newer one for testing.
    • In some cases, you may have 2 different template for testing two very different page, normally in two different Personas.
  6. Trial - Expermients are run at the Trial level. A trial is a test of a subject against a group of visitors (referred to as an audience). The audience can be tested as a whole or you can divide your audience into Personas if you want to optimize based on some common trait.


  7. Personas - A persona is a group of visitors with common traits participating in a trial. A trial can have any number of Personas. Each trial is created with a default persona named "All Visitors", and other Personas can be added subsequently. Personas (except for the trial default persona) can be assigned a criteria that determines which visitors become a member of that persona. A few things about Personas:
    • Our Best Practice dictate that we get an STP for testing:
      • We create a QA Persona for testing.
      • We create a Blank Wave based the Blank Template - one creative only - and make it the current wave om the All Other Visitors Persona.
      • We then test the Template using the QA Persona, then push the content of Wave 1 to the All Other Visitors Persona when we are ready to push the test to live customers.
    • When you set up multiple Personas, know that if the criteria are NOT mutually exclusive, you will need to set a priority in the Persona to decide which one has higher priority if both are selected as true. Be wary of this possible collision.
  8. Wave - A wave is a test of a specified set of creatives run against a persona of visitors within a trial. The creatives tested are chosen based on an Optimal Experimental Design given the variables and the values being tested.
  9. Sector - A sector is a group of creatives within a wave. Each wave must have at least one sector. Often, each wave has two sectors. One sector is named "Control" and contains the control creative for the subject. The second sector is named "Challengers" and contains creatives selected by an Optimal Design. The response rates of these creatives will be compared to the control. After the first wave is complete a third sector named "Winners from Wave X" is added that contains the best creatives from the previous wave. It's important that you establish these baselines so you can take into consideration changes in your visitor base that may influence results.

Testing and QA - things to think about

  1. Anytime you make changes to a template, you must Refresh Creatives to see the changes on the served content (the Creatives going out to Customers).
    • When you create a new Wave and you activate it, at that moment it takes a snapshot of the template.
      • The creatives you defined will be saved to our Optimost Creative Servers with the template's code as it stands at that moment.
      • The creatives will be served as such going forward, with only the session identifier changing on new page views.
    • If you make subsequent changes to that Template, those changes will not be reflected in the current Creatives unless you Refresh Creatives.
    • You can Refresh at the Template OR Wave level, depending on what you are doing. For example:
      • If you have a template in different waves in multiple Personas,
        • If you are testing something in QA and don't want to effect the template in another Persona, refresh at the wave level.
        • If you want changes to be reflected in all waves, then Refresh at the Template level.
      • If you have a template in a wave in only 1 Persona, it does not matter, you can Refresh at the Wave or Template level.

  2. More generally, many changes at the Subject level require you to Refresh Creatives. Examples include (circled in red in the diagram below):



    1. Changes to the Template, as noted in #1.
    2. Any changes to the Activity Tracking OR Stickiness Cookie settings
      • These are saved as part of the creative
    3. Changes to the clickthru domains (at the Subject OR Portfolio level)
    4. Changes to the Page View Attributes
    5. Changes to Modules
      • A new one is added
      • The name of an existing module is changed.

  3. Examples of Changes you DO NOT need to Refresh Creatives for (circled in blue in the diagrams above and below):



    1. Changes to Failover Copy
    2. Trial Level changes such as:
      • Adding or removing Personas
      • Changes to the Persona Criteria
      • Changing the Current Wave.

  4. When you are creating a Wave, if it is an experimental design on an MVT, you must check the box "Sector is supported by an experimental design".



    • Without this being checked, you cannot see the Sector variable report, which for an MVT is bad not to have, and you will likely have to rerun the wave if you want it to appear in the reporting. Very bad, make sure to check it if need be.
    • And if it is not an MVT but an A/B/n test, the box should be unchecked because than you would have a sector variable report available (that should not be looked at for an A/B/n test) that may confuse someone looking at the Wave.

  5. The different Stages of a Trial
    1. Design - You have at least one Portfolio that does not have a "current wave".
      • All new Trials start this way.
      • Content is served to any Personas with a current wave.
    2. QA - Each Portfolio has a "Current Wave", but you have not "Launched" the Trial to make it "Live".
      • QA and Live Trials both serve content and collect data in each Persona in the exact same manner.
      • A QA Trial can still have Personas added or removed.
    3. Live - Each Portfolio has a "Current Wave", and you have "Launched" the Trial to make it "Live".
      • Live in Optimost speak means we have an experiment that will not change in terms of Personas, and has been sufficiently vetted to ensure we are ready to run the experiment.
      • Best practices tells us to wait until the experiment has been running a bit to launch the Trial to Live. Once you do that, you can't make Persona changes. I cannot stress how important this is.
    4. Fallback - When we progress from one Trial to the next - this is done by launching the new Trial - the previous Trial become Fallback.
  6. How to handle going from Trial n to Trial n+1
    1. Trial n must be "Live", and Trial n+1 must be in "QA".
    2. Now that each Trial will have its own placement, which is the Trial code URL.
      • This means you can serve both Trials at once if the customer deploys the different URLs in different environments
      • We originally intended to do this with Live and QA Trials, but customers generally don't like doing that.
    3. To progress to Trial n+1, you hit "Launch Trial", and it will ask you if you want to use the placement from this Trial or Trial n.
      • In most cases, we want to continue to use the placement that exists on the customer's page, which is the placement for Trial n.
