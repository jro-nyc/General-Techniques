Background

One of our partners, Digital River, tends to run a bunch of experiments simultaneously. Because of this, they are running into the problem of running "out" of First-party cookies with IE 6 and some IE 7 users - ones with browsers with a 20 cookie per domain limit. Since we normally drop 2 cookies per Subject/Trial Code call, if you are running 5 tests on a site, you are using 10 cookies or half of your available pool, and if the site also has more than 10 cookies they want to drop - not that unusual - then we have a problem. DR was going over by 1 or 2 cookies, so their solution was to make the stickiness cookie cross-domain, and their issue was taken care of. But, ideally, we would like to have a more scalable solution. This entry is about such a solution

Activity Tracking and Stickiness

In order to do either Activity Tracking or Stickiness, we need to have a way to know what creative a visitor saw so that:

  1. If that visitor returns to the page or refreshes the page, we can show them the same creative again.
  2. If that visitor progresses down the funnel and gets to a "success" page, we can associate that success with a specific visit.

The first scenario describes "Stickiness", the second "Activity Tracking". Back in the day, these values where often explicitly stored on our customer's backend and sent back to us when visitor returned or when a success occured. Over time, it was decided that to ease our implementation process, we should drop these values as "cookies".

Since we are doing this using cookies, the developers decided that you needed separate parameters since, although they have the same value when a visitor arrives at our page, the "lifetime" of these values may differ. In most cases, we set the stickiness for 30 days, and the Activity Trakcing for 1 day.

At the present, over 99% of our customers use cookies for tracking.

This code includes two functions:

  1. opCreativeSetCookieA - to set the cookie's name value pair, domain, and expiration
  2. opCreativeGetDocumentSLD - to get the document domain - assumed to be a 2 word base domain - for opCreativeSetCookieA

and then calls to those fuctions to set the two cookies. Notice the cookie values are indentical - the Session Identifier for that page view - and the values only differ by the last characters - gum for the Stickiness cookie and liid for the Activity Tracking cookie.

Getting the cookies

To get the cookies, the Page Code pulls them out by finding all First-party cookies that start with "op" and attaches them to the Trial code URL, so that the URL will look something like:

https://by.essl.optimost.com/by/trial/0/p/registrationpage.ab4/1/content.js?op0registrationpagegum=a00401z00625ali0n96n4e851&op0registrationpageliid=a00401z00625ali0n96n4e851

Server context and the Solution

Understand that when the visitor's browser makes a call to our servers to serve a creative, the cookie values are passed back as Query String parameters. The fact that this value was stored as a single cookie is irrelevant to the server, the only thing that needs to happen is that it needs to be passed in the URL. As I mentioned before, back in the day the values were stored on the Customer's servers and then attached to the URL by a back-end process. The Server didn't care/know then and doesn't care/know now how it is stored. (caveat - if you use cross domain tracking, the Server does care because it has to look for those in the header, not the URL's Query string)

So we could just as easily store all the First-party cookies in a single cookie on the Visitor's PC, parse out all the individual name-value pairs, and send it back in the Trial code URL as separate Query string parameters.

Issues with the Solution

This solution has to be done using purely front-end code, with no extra development to the Console by the Development team to accomodate it - ie they are not changing the code that drops the Activity Tracking and Stickiness cookies to suddenly drop a "multi-cookie". With that in mind, here are the major issues.

  1. There will have to be a "modification" to the Page code to allow us to read and write the "multi-cookie" cookies, and parse out the individual name-value pairs to send back in the Trial Code URL.
  2. We must account for the individual time stamp/expiration of the individual cookies inside the "multi-cookie"
  3. We should delete the standard Activity Tracking and Stickiness cookies since the code will continue to drop them.
  4. We will need new counter code to parse out the "multi-cookie" for the individual name-value pairs to send back in the Counter Code URL.
  5. Although this makes it less likely we go over the number of cookie limit for a browser domain, we could still go over the cookie size limit. In 2007 documentation, IE says you each domain can have up to 20 cookies with a total size of 4096 bytes, which means one 4096 byte cookie or 20 cookies that total 4096 bytes. Be conscious of this in light of:
    • How many experiments we will be running at once, which effects how large our multi cookie will get
    • The number and size of the cookies the customer is presently dropping in their domain.

Code for "multi-cookie"

I had to create modified Page Code and modified Counter Code

Modified Page Code and example calls

This code does the following:

  1. It takes the Page Code and adds two functions:
    1. getMultiCookie(type) - Executed first, it looks for an existing cookie whose name starts with "op" and whose value starts with "optmulti". Parses this cookie and adds the name-value pairs whose time stamp has not expired to the Optimost.C object, and resaves the multi-cookie with all the name-value pairs whose time stamp has not expired. The parmeters are:
      1. If the parameter type is "add", it returns the "multi-cookie" value without checking the timestamp for each name-value pair..
      2. If the parameter type is "get", it returns the unexpired "multi-cookie" name-value pairs with thieir individual timestamps.
    2. setMultiCookie(add,nm,v,e,d,ckn) - Calls getMultiCookie() to get the multi cookie and now adds the new cookie name-value pair plus time stamp that is passed in. The parmeters are:
      1. add - boolean, whether you want to add the drop the cookie as a first-party cookie in addition to adding it to the multi-cookie.
      2. nm - the cookie name
      3. v - the cookie value
      4. e - the cookie expiration time in seconds
      5. d - the domain
      6. ckn - the name for the "multi-cookie". If a null, defaults to "opmulti001"

http://asp.optimost.com/webtest/jim/wiki/opt_newMultiAll.html

Look at the "Net" tab, and look at the "Params" tab in the two JS files. You will see all the cookie values - 15 in all - being sent back to the Optimost servers set there. Next, load:

http://asp.optimost.com/webtest/jim/wiki/opt_newCounterCode.html

and look a the single JS file in the "Net" tab and its "Params" tab. It should also have all the cookies we set. Refresh this file every 10 seconds or so, and check the params. You will see that after a minute - the time stamp I set for 13 of the cookies - only the op001testCookieliid and op001testCookiexyz will remain, and after 2 minutes, only the op001testCookieliid will remain.

Implementation

In order to implement this code, you would need to do the following:

  1. Add the two new functions - getMultiCookie(type) and setMultiCookie(add,nm,v,e,d,ckn) - to the page code, either by deploying new Page Code OR adding the functions to the Optimost Object at the top of the Setup block for the template.
  2. Just below the functions - or at the top of the setup block is the Page Code is modified - you would now drop the "multi-cookie" and remove the one dropped by the Trial code. So for the example we used above: 
function opCreativeSetCookieA(n, v, d, e){var de = new Date;de.setTime(de.getTime() + e * 1000);
document.cookie = n + "=" + escape(v) + ((e==null) ? "" : ("; expires=" + 
de.toGMTString())) + "; path=/" + ((d==null) ? "" : (";domain=" + d));}

function opCreativeGetDocumentSLD(){var sld = document.domain;var dp = sld.split(".");var l = dp.length;
if (l < 2) sld = null;
else if (!isNaN(dp[l-1]) && !isNaN(dp[l-2])) sld = null;else sld = "." + dp[l-2] + "." + dp[l-1];return sld;}

opCreativeSetCookieA("op0registrationpageliid", "a00401z00625ali0n96n4e851", opCreativeGetDocumentSLD(), 604800);

You would need the following in the setup block: 

optimost.setMultiCookie(false,"op0registrationpagegum","a00401z00625ali0n96n4e851",2592000,optimost.SLD(),null);
optimost.setMultiCookie(false,"op0registrationpageliid","a00401z00625ali0n96n4e851",604800,optimost.SLD(),null);
