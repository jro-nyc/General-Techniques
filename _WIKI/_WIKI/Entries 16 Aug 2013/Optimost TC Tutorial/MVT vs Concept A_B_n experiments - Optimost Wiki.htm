Concept test (A/B/n) vs MVT experiments

Optimost is the premier multivariable testing (MVT) provider, and has been since we pioneered the space with a company launch in 2001. An MVT is a set of related, connected variables that work together to form a test of a visitor’s user experience. Variables can vary from a simple headline change to complex page layout changes. Our MVTs use orthogonal experimental designs to run experiments efficiently and arrive at quantifiable results as quickly as possible.
Since an MVT separates the page into test areas, we retain the original structure of the page – we make are “surgical” modifications to each test area by manipulation the DOM1 via javascript or CSS. This allows us to run experiments on dynamic pages2 by setting up experiments to test "around" the changing content.
For most Concept A/B/n tests, however, we have to replace the entire content of the control page with a challenger version of the page. Unlike an MVT, Concept A/B/n versions can make changes to the page that modify the structure of the page, when such changes cannot be created by manipulation of the DOM. In replacing the content, we lose the previous content of the page, and if the page is dynamic, retaining any dynamic elements on the page may or may not be possible.

1DOM – Document Object Model, a standard for representing and interacting with objects on a web page.
2Dynamic Pages – Pages with changing content; i.e. a page can show different products (an ipod, towels, chairs, etc), a page that looks different for logged in users, a page with articles that change as world events happen, etc.

An MVT example

Figure 1 shows the Optimost Home Page. As an example, let's say we are running an MVT on the areas specified. For variables A, B, C, and D, we test the content of each area. For variable E, we test the location of areas A-D. For F, we test the content of the bottom 4 sections as a single variable.
To make these changes, we simply examine the DOM, find the different test areas, and using either javascript or CSS, make changes to each area. In this MVT:

  1. Areas A-D are inside a table – one column (<td>) for A and one column for B-D.
    • Area A is a flash.
    • Areas B-D are images inside an <a> tag, each inside of a separate <div>.
  2. Variable E is not an area, but a layout change, as in:
    • The ability to move area A to the right and areas B-D to the left
    • The ability to shuffle areas B-D above and below each other.
  3. Area F is inside a table, with one column (<td>) for each of the 4 sections.
    • Each column contains a <table> with the content.

Figure 2

Figure 2

Figure 2 shows the DOM structure using Firebug. Each of the test areas can be found using javascript, So for this MVT, the changes to these 5 areas and the one basic layout change can be made while retaining the original structure of the page.

A Concept A/B/n example

Next, let’s say we wanted to test a very different version of the Home Page.

Figure 3

Figure 3

Figure 3 shows a version with the following differences:

  1. No content above the top navigation bar.
  2. A slightly modified top navigation bar.
  3. New content between the top navigation and main content area.
  4. The main content area has:
    • Different layout presentation and style.
    • Different specific content.
  5. A new right-side column with a testimonial and upcoming events.

Looking at the differences, we can see two major issues:

  1. This "B" version, when compared to the original "A" version, cannot be broken into test areas as we did in the MVT.
  2. If we look at the DOM structure of the page – as we see in Figure 4 – we see a very different structure and layout for the nodes on the page.
Figure 4

Figure 4

So, given these two factors – either of which by itself would make DOM modifications impossible – in this case, we will have to do an HTML substitution for the entire test area, which in this case is the entire page.

Dynamic content example

For testing a Landing Page like the Optimost Home Page, there is generally not a lot of content that changes in real time. Things like the upcoming events may be updated on a weekly basis or so, but even in the case where the events are not updated, although it will make the page less useful, it will not "break" the functionality of the page. Here we will dive a little deeper to show the two most common types of dynamic pages we test and explain why running an MVT on such pages is relatively simple BUT running Concept A/B/n may be at best much more difficult, or at worst nearly impossible .

Product Pages

Product pages, being an integral part of the revenue generating process for e-commerce sites, are prime targets for MVT testing. Figure 5 shows an example page.

Figure 5

Figure 5

This product page is selling a book, and all the books have common elements, dynamic elements, and “business rules” associated with the content. For example:

  1. There are a number of dynamic elements:
    • Book specifics – name of the book, its image, the name of the author.
    • The price for the different types of offering.
    • Customer rating information.
  2. Common elements include:
    • The position and style presentation – fonts, colors, sizes – of the content.
    • The types of buttons.
  3. There are a number of "business rules":
    • "Customers who bought this also bought" recommendation section may have a different number of items, or even not exist, depending on the book you selected.
    • They may or may not have used books.
    • The book may or may not be in stock.
    • All books may not be available in all the different formats

Therefore, on this page we could test, in an MVT:

  1. The position and style – fonts, colors, sizes – of the content.
  2. The appearance or copy of the buttons.
  3. The layout of elements – moving them around in ways that maintain the structural integrity of the page
  4. The existence or non-existence of certain elements.

Now let's say the creative team decided to go crazy with a brand new site design, and test the design below as the challenger “B” version.

Figure 6

Figure 6

Figure 6 shows a page that clearly has a completely different structure. We would not be able to "surgically" modify the DOM of the original page to produce this one, so we would have to do an HTML replacement. However, because this is a dynamic page, we would have to extract all the dynamic elements from the original and place them in the B version. This process would be extremely difficult and time consuming, and for that reason we avoid doing Concept Tests on these types of pages.

Form Pages

Here we have the original Form page for a website.

Figure 7

Figure 7

Their creative team has decided that they want to test an alternate version, and provided the HTML for the following version3:

Figure 8

Figure 8

Looking at the challenger, the structure is completely different, so we must do HTML substitution, not DOM manipulation. But this is a form, and it may have dynamic elements, which including:

  1. Hidden inputs that pass session specific information
  2. Front-end form validation built into the page – usually AJAX based.

This form page contains much less dynamic content than a product page, but with an important complication – unlike a product page, a form page needs to successfully validate and post data to the customer’s back end. So when you are doing an HTML replacement now you have to copy/store all the pertinent state information from the original page and add it to the replacement version so that it looks exactly like the original page to both the back-end (i.e. when you post data, it is exactly what the back-end is expecting) and to any validation instrumentation on the page.

3Best practices require the Customer provide a working version of the page for Concept tests. Receiving HTML only will require a significant increase in the time-line. We are using this example to illustrate complications associated with Concept A/B/n tests containing dynamic content.

Summary

Optimost can do both MVT and Concept A/B/n experiments – MVT using DOM manipulation, Concept A/B/n using HTML substitution. In the instance where the experiment contains dynamic content, we strongly recommend that it be an MVT. If the Customer does insist on doing a Concept A/B/n experiment on a page with dynamic content, know that our standard timelines do not apply to such experiments, and the time needed for implementation it may be significantly greater than an MVT.
