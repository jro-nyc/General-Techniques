Why Server to Server?

Most of our work is done with modular templates. The clients puts page code and trial code on the page, wraps the test area(s) with module code, and we test the page. This covers over 90% of the experiments we do. However, instead of going directly between the Optimost Servers and the Visitor's Browser, we may want/need to send the creative information directly to the Customer's servers. Reasons could include:

  1. We are running an Email test, and since emails cannot use javascript, a modular implementation will not work.
  2. The customer's page has very sensitive private information, and they do not want the appearance that a third-party could have access to that information, so they eliminate the need for javascript code in the visitor's browser by first getting the creatives from Optimost, then rendering the web page with that creative information from their servers.
  3. The customer's security department has a "no third-party content" policy, so we cannot put Trial code on the page. So, as #2, they eliminate the need for javascript code in the visitor's browser by first getting the creatives from Optimost, then rendering the web page with that creative information from their servers.

Modular implementation

Over 90% of our test are modular implementations, and here is the basic flow of those tests.

  1. Visitor comes to page under test, and calls customer’s servers.
  2. Customer's servers present the HTML, CSS, and javascript for that page to the Visitor's Browers
  3. With the content in hand, the browser begins executing the code. When it finds calls for external content, it calls the servers in question to get that additional content.
  4. The browser reaches the Optimost Trial code call. This includes a call to our servers, so the browser calls the Optimost servers, and ….
  5. Our Servers return the code necessary to implement this test. This includes all the modules defined for this experiment.
  6. The browser continues to execute the content. Later in the page, if it arrives at Optimost module code, it will execute the module code, which will make the changes to that test area.
  7. The browser proceeds and eventually gets to the end of the content, and then finishes its execution.

Server To Server - HTML template

If we decide to do Server to Server for an HTML template, here is its basic flow.

  1. Visitor comes to page under test, and calls customer's servers.
  2. Customer’s servers recognize that this page is their test page, and they call the Optimost servers to get the creative for the test. In the call they will include any stickiness and activity tracking information that tells Optimost which creative, if any, they visitor had previously seen
  3. With this information from the customer's servers, Optimost gets the creative of interest and sends it back to the Customer's servers. This is normally sent using XML, which can be anything from a an HTML page with XML wrappers around it, or a pure XML version of the page under test with a instruction set defined by the customer.
  4. The Customer's servers take this XML and do any necessary processing to construct the HTML page needed for this creative
  5. Customer’s servers present the HTML, CSS, and javascript for that page to the Visitor's Browers, including the code for the area we are testing. They will be responsible for dropping the stickiness and activity tracking cookies.
  6. With the content in hand, the browser begins executing the code. When it finds calls for external content, it calls the servers in question to get that additional content. The browser proceeds until it gets to the end of the content, and then finishes its execution.

Server to Server, Email template

First, the Email vendor and Optimost need to decide on the process to be used. We can either do it as a Server to Server XML application or as pre-constructed Email templates. Please take a look at:

page:toBeAddedLater

for more information on the difference.

Let's look at the two possibilities within the Server to Server app

Standard test - Email vendor hits our servers once per email to be sent

In a standard Optimost experiment, each “page view” – in this case email – requires a call to the Optimost servers. With that scenario in mind:

  1. Email vendor decides to send an email, so their Server call the Optimost Servers.
  2. Optimost returns a randomly-selected email creative from the experiment/wave that is running. If necessary, Optimost had built the creative with placeholders for dynamic content to be added by Email vendor.
  3. Email vendor takes received email and adds any necessary dynamic content.
  4. Email vendor delivers email to intended recipient.
  5. Email vendor repeats process for each email to be sent.
  6. Optimost analyzes results in normal fashion.

Image:EmailStandardTest.gif

"Once per creative" test - Email vendor hits our servers once per creative to be sent

In an alternate scenario, the Email vendor only wants to call the Optimost servers once per creative in the experiment, then they will take each of the creatives and distribute them evenly amongst the intended email recipients. With that scenario in mind:

  1. Optimost manually provides Email vendor with list of creatives in current experiment/wave. Let's assume there are N creatives
  2. Email vendor calls Optimost Server and passes the creative number for the first creative.
  3. Optimost returns the specific-defined creative from the experiment/wave that is running. If necessary, Optimost had built the creative with placeholders for dynamic content to be added by Email vendor.
  4. Email vendor takes received creative and stores it as one of N creatives. Email vendor can add any necessary dynamic content that is best added at this point (Email vendor discretion).
  5. Email Vendor repeats steps 2 to 5 for all N creatives to be able to store all creatives.
  6. At some later time – could be seconds later, could be hours, days – email vendor randomly picks 1 of N creatives to send to a specific email recipient. May add any necessary dynamic content that is recipient-specific at this point.
  7. Email vendor delivers email to intended recipient.
  8. Email vendor repeats steps 6 to 7 for each email recipients.
  9. Optimost will do analysis off-line since single page view send multiple times and standard analysis will get confused.

Creating the XML template

If we do Server to Server, then XML is the template type. In this instance XML is used to store and transmit data, and there are a few approaches:

Content as pure XML

In this example, we send the individual elements not as HTML, but as XML data, For example, let's say we have 3 images we are testing with one link. There are two ways to do this:

Passing URLs

Here we send the images as URLs, and the clickthru as a click. Here is some sample XML. 

<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>http://es.optimost.com/es/0/c/0/u/A1.gif</AREA_A>
		<AREA_B>http://es.optimost.com/es/0/c/0/u/B3.gif</AREA_B>
		<AREA_C>http://es.optimost.com/es/0/c/0/u/C2.gif</AREA_C>
		<CLICK_TRACKING>http://by.optimost.com/go/0-20/a0fb0i4e249i1ei30rb4db/</CLICK_TRACKING>
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

Here the server needs to know where to plug in the individual URLs into the HTML that the customers will see.

Passing text instructions

Here we send text instructions indicating which image to use, and the clickthru as a click. Here is some sample XML. 

<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>value 1</AREA_A>
		<AREA_B>value 3</AREA_B>
		<AREA_C>value 2</AREA_C>
		<CLICK_TRACKING>http://by.optimost.com/go/0-20/a0fb0i4e249i1ei30rb4db/</CLICK_TRACKING>
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

In this case, the data we are passing needs to first be converted into an actual URL (or whatever element it happens to be), then added into the HTML of the page as appropriate. For organizations that want full control of all the content generated, this is the best approach.

Content as HTML wrapped in XML

In this example, we send the elements as HTML. For example, let's say we have 3 images we are testing with one link. Here we send the images as URLs, and the clickthru as a click. Here is some sample XML. 

<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_ABC_HTML>
			<table>
				<tr>
					<td><img src="http://es.optimost.com/es/0/c/0/u/A1.gif" /></td>
					<td><a href="http://by.optimost.com/go/0-20/a0fb0i4e249i1ei30rb4db/">
						<img src="http://es.optimost.com/es/0/c/0/u/B1.gif" />
					</a></td>
					<td><img src="http://es.optimost.com/es/0/c/0/u/C2.gif" /></td>
				</tr>
			</table>
		</AREA_ABC_HTML>
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

Note that XML gets validated with specific syntax rules, so that the HTML inside the XML must open, close, and nest tags properly, as well only use syntax characters for markup delineation.

Content as HTML containing javascript wrapped in XML

In this example, we add inline javascipt to the HTML example above. 

<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_ABC_HTML>
			<table>
				<tr>
					<td><img src="http://es.optimost.com/es/0/c/0/u/A1.gif" /></td>
					<td><a href="http://by.optimost.com/go/0-20/a0fb0i4e249i1ei30rb4db/">
						<img src="http://es.optimost.com/es/0/c/0/u/B1.gif" />
					</a></td>
					<td><img src="http://es.optimost.com/es/0/c/0/u/C2.gif" /></td>
				</tr>
			</table>
			<script type="text/javascript"><![CDATA[
				if((aVal=='3')||(aVal=='5'))document.write('<img src="../images/optimost/areaA4'
 				+ aVal + (aVal=='3'?'.gif':'.jpg') +'" border="0" id="section_a" alt="" />')
			]]></script>	
		</AREA_ABC_HTML>
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

Since the content of the script block does not follow the well-formed XML rules, so it must be escaped inside a CDATA block.

Tracking - Cookies and an XML template

for dropping the Stickiness and Activity Tracking cookies, we have 2 options:

  1. Have Optimost pass the cookie-dropping javascript code in the XML CDATA script block.
  2. Have Optimost pass in the cookie name-value pair as XML data and the Client either:
    1. Drops the cookies using Server Side code
    2. Drops the cookies using javascript in the web browser

Passing cookies via XML

Here we pass the Activity Tracking cookie and Stickiness cookie parameters via XML. We pass in:

  1. The name of the cookie
  2. The value for the cookie
  3. The cookie expiration time, in seconds. Usually set to 1-day for Activity Tracking and 30-days for Stickiness. 
<?xml version="1.0" ?>
<OPTIMOST_RESPONSE>
  <OPTIMOST_CONTENT>
    <COOKIES>
      <ACTIVITY>
        <NAME>op2turnvexmlliid</NAME>
        <VALUE>a0s70xd3wm26b9i0pk43db0e3</VALUE>
        <TIME>86400</TIME>
      </ACTIVITY>
      <STICKY>
        <NAME>op2turnvexmlgum</NAME>
        <VALUE>a0s70xd3wm26b9i0pk43db0e3</VALUE>
        <TIME>2592000</TIME>
      </STICKY>
    </COOKIES>
  </OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

With this data, the client would drop the cookies from the Server-side or in the browser (using javascript).

Passing cookies via javascript in the XML

Optionally, we can pass the javascript in the XML, and the Client would have to place this code in the HTML. 

<?xml version="1.0" ?>
<OPTIMOST_RESPONSE>
    <OPTIMOST_CONTENT>
        <COOKIES>
	<script type="text/javascript"><![CDATA[
            function opCreativeSetCookieA(n, v, d, e){var de = new Date;de.setTime(de.getTime()
            + e * 1000);document.cookie = n + "=" + escape(v)
            + ((e==null) ? "" : ("; expires=" + de.toGMTString())) + "; path=/" +
            ((d==null) ? "" : (";domain=" + d));}
            function opCreativeGetDocumentSLD(){var sld = document.domain;
            var dp = sld.split(".");var l = dp.length;if (l < 2) sld = null;
            else if (!isNaN(dp[l-1]) && !isNaN(dp[l-2])) sld = null;
            else sld = "." + dp[l-2] + "." + dp[l-1];return sld;}
            opCreativeSetCookieA("op2turnvexmlgum", "a0s70xd3wm26b9i0pk43db0e3", 
                        opCreativeGetDocumentSLD(), 604800);
            opCreativeSetCookieA("op2turnvexmlliid", "a0s70xd3wm26b9i0pk43db0e3", 
                        opCreativeGetDocumentSLD(), 86400);
	]]></script>        
        </COOKIES>
    </OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

In this scenario, the customer places the javascript on the page as inline code, and it will drops the cookies.
