Introduction

Flash is a plug-in that is commonly used to create animated and interactive content on Web pages. Flash may appear as off-site advertising banners, single elements on-site (flash modules inside an HTML page), or as entire Web page or Web site programmed in Flash. It is added as an HTML embedded object element, which is also used to embed applications such as video, audio, or java applets into web pages.

File formats associated with flash include:

  • .swf: The files that are completed, compiled, and published. They cannot be edited.
  • .fla: The source material for the Flash application.
  • .as: The ActionScript source code.

Here is a sample object element containing flash that has a URL of multiVarTest.swf: 

<object classid="clsid:D27CDB6E-AE6D-11cf-96B8-444553540000" 
	codebase="http://download.macromedia.com/pub/shockwave/cabs/flash/swflash.cab#version=6,0,0,0" width="700" height="300">
	<param name="movie" value="multiVarTest.swf">
	<param name="quality" value="high">
	<param name="bgcolor" value="#336699">
	<EMBED src="multiVarTest.swf" menu=false quality=high WIDTH="700" HEIGHT="300" ALIGN="" TYPE="application/x-shockwave-flash" 
	PLUGINSPAGE="http://www.macromedia.com/go/getflashplayer"></EMBED>
</object>

Testing Flash with Optimost

On-Site/Module

We're testing an entire page or a single module on the page. With Flash testing, the Optimost platform creates a "Flash Utilities Code" in addition to the Page Code. The Flash Utilities Code should load after the Page Code but before the page content; it will tell the page what to display. Instead of programming JavaScript instructions, we instruct the Flash player to display a certain page instead of displaying it ourselves. The client is responsible for programming the Flash content to display the variable values. We can instruct the Flash module to display content via XML calls, which the client will generally be using to feed content to the Flash display anyway.

When the Flash content loads it calls the Optimost Page Code and Flash Utilities Code to obtain the URL needed for the XML call to the Optimost servers.

Off-Site Testing

We may not have access to the ad server, so we will not be able to insert Page Code or Trial Code. Instead, the client must hard-code the XML call to the Optimost platform into the Flash module so that the call will be made regardless of which page the ad is served to. The Flash module will make an XML call to the Optimost platform, which will return an XML command to the Flash module to tell it what to display.

Example: Changing Flash content via XML

In this example, the Flash content to be manipulated is a variable with three values; the Optimost platform must pass the image URLs to the Flash module.

This is the Optimost Template: 

<?xml version="1.0" ?> 
<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>{{$areaA}}</AREA_A>    
		<AREA_B>{{$areaB}}</AREA_B>    
		<AREA_C>{{$areaC}}</AREA_C>    
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

Since the values are simply image URLs, the response returned by the Optimost server is: 

<?xml version="1.0" ?> 
<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>http://es.optimost.com/es/308/c/2/u/14465_btn_apply_now_burgundy_plain.gif</AREA_A> 
		<AREA_B>http://es.optimost.com/es/308/c/3/u/v2.jpg</AREA_B> 
		<AREA_C>http://es.optimost.com/es/308/c/3/u/prev_active.gif</AREA_C> 
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

Differences from Normal Implementations

This requires a lot more programming resources from the client than the normal Optimost implementation. They must be prepared to do a significant amount of Flash programming and this may slow down the timeline significantly, depending on client resources.

  • The majority of the work is performed by the client.

Optimost creates a template shell, but the client must program the Flash to:

  • initialize the XML feed
  • drop cookies for proper tracking
  • track clickthrus
  • read in the various values for each variable and execute the command provided by Optimost

Optimost must perform the responsibilities and deliver the following:

  • Create the template
  • Deliver the CSI code
    • Page Code
    • Flash Utilities Code
  • CSI Guide must include instructions on how to set cookies and implement clickthru tracking/clicktags
  • Deliver cookie code
  • Counters


Things to note

Be sure you specify what the params and attributes are for flash assets, there can be times where IE7 blocks the flash file if certain things are missing. Missing the classid for example will activate the IE7's "Click here to activate and use control" message before you can actually use the flash. So beware!

If links doesn't work

Sometimes this piece of code works
<param value="always" name="allowscriptaccess"/>


Adding additional flash objects into the same HTML page
Note the classid attribute of the object tag.  Firefox (and Opera and Safari) seems to care about this (or, perhaps I should say, IE doesn't care about this) and renders user actions moot. Normally the classid comes with the flash object and is not an issue.  If you use one flash object as a template to create another (for example, moving the first across the page and replacing it with a new flash object), this classid attribute has to be unique !

Tracking Considerations

Can first-party and third-party cookies be used for activity tracking and stickiness for XML Flash Templates? Tracking Clicks (Clicktags)

  • We pass the clickTag value via an XML stream. Just need to coordinate how that is received with the customer.
  • Dropping the activity tracking and stickiness cookie. Again, we will have to pass the value in via XML, but then have to write some Script code to drop the cookie.
  • Retrieving the stickiness cookie. Script code will have to look for this cookie, and if it exists, pass it back to Optimost in the call from the flash.
  • Triggering any internal "events" that do not redirect. This is applies to events we use binary counters to track - say clicking on a button to see another part of a the flash movie, and the client wants to know if this happened. Again, we would need to write Script code for the flash to do this

QA Considerations

  1. Cross-Browser issues

Testing in Flash does not require cross-browser testing.

  1. Pinning creatives
  2. QA environment
  3. Client is new to process and methodology. The client may not take all combinations or constraints into consideration because they're not familiar with multivariable testing. Be mindful of this during planning and preparation and during QA.
  4. Because of the client's unfamiliarity with the testing, make sure your QA wave is larger than usual and covers more combinations than usual. We run into both functional and aesthetic issues with untested combinations. QA here has no access to XML information fed by client to QA server, so the client must be made aware of exactly of what they must be checking.
  5. Clickthru tracking/debugging differs from HTML implementation.

XML IE6 Bug - Please Note

In an implementation, the Flash will be making a cross-domain call. That's fine, Optimost has a file that tell the Flash we are an "OK" domain. The IE6 issue has to do with GZIP compression. At the subject level, under Change subject options, you need to select never and you should be fine. If there are issues still - secure/non-secure issues may crop up - you need to coordinate with the Flash developers to show default content if the XML call is not succesful.
