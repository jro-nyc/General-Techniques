Background

We were trying to do some ad testing with one of our customers. Their creative agency, Carat, was responsible for creating the Flash content. We explained how they needed to make a flash movie that could be "programmed" by parameters passed in by Optimost. Because the content was being served by Double-click, it looked like using our Page Code/Trial Code implementation would be problematic. So they suggested us passing in the content via XML... HMMM.

Hey, we can do that!

How to pass data in via XML

It's actually quite easy. To prove the concept, Carat created a Flash movie example that expected values to be sent via an XML feed, and would use those values as the URLs for 3 images within the Flash movie. On the Optimost end, we created an XML template that had 3 variables - the 3 images inside the flash - each with a number of values.

SO we sent them the URL for the XML feed, told them what the XML data set would look like, and they did the rest. Here's how the Template looked: 

<?xml version="1.0" ?> 
<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>{{$areaA}}</AREA_A>    
		<AREA_B>{{$areaB}}</AREA_B>    
		<AREA_C>{{$areaC}}</AREA_C>    
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

Where the variable values are just URLs to images. So for A=2, B=1, C=3, the XML feed returns: 

<?xml version="1.0" ?> 
<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>http://es.optimost.com/es/308/c/2/u/14465_btn_apply_now_burgundy_plain.gif</AREA_A> 
		<AREA_B>http://es.optimost.com/es/308/c/3/u/v2.jpg</AREA_B> 
		<AREA_C>http://es.optimost.com/es/308/c/3/u/prev_active.gif</AREA_C> 
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

The XML URL used is:

http://by.optimost.com/trial/2/p/banneradxmlexample.d31/456/content.xml

feel free to stick it in the browser and hit "refresh" to see what comes back from our Servers.

Issues with using XML implementation for flash

What remains is how to do the tracking. And here we have the following issues:

  1. Tracking Clicks should not be any different, but now we pass the clickTag value via an XML stream. Just need to coordinate how that is received with the customer. You can click on this URL and see what the XML looks like: http://by.optimost.com/trial/2/p/banneradxmlexample.d31/456/content.xml?size=1
  2. Dropping the activity tracking and stickiness cookie. Again, we will have to pass the value in via XML, but then have to write some Script code to drop the cookie. See Flash and XML - Creating a Flash-only Template using XML
  3. Retrieving the stickiness cookie. Script code will have to look for this cookie, and if it exists, pass it back to Optimost in the call from the flash. Again, see Flash and XML - Creating a Flash-only Template using XML
  4. Triggering any internal "events" that do not redirect. This is applies to events we use binary counters to track - say clicking on a button to see another part of a the flash movie, and the client wants to know if this happened. Again, we would need to write Script code for the flash to do this.

What will the "Script code" look like

This could be flash "Live Script" code written inside the flash, or javascript functions we add to the page that we program the flash to call.

XML IE6 Bug - Please Note

In an implementation, the Flash will be making a cross-domain call. That's fine, Optimost has a file that tell the Flash we are an "OK" domain. The IE6 issue has to do with GZIP compression. At the subject level, under Change subject options, you need to select never and you should be fine. If there are issues still - secure/non-secure issues may crop up - you need to coordinate with the Flash developers to show default content if the XML call is not succesful.
