Flash and Optimost - Implementation concerns for standard Modular implementations

I found that clients are extremely interested in testing flash however it always comes down to a matter of resources, as most of the work has to be done on their end to set this up.

As an object, there is unfortunately no way to "peer" into the flash the way we can "peer" into other HTML elements. So we cannot "track" flash behavior using the normal means we use to track behavior on HTML elements such as links, images, forms, etc. The events that would be of interest - clicking on a flash "link", submitting a flash "form", initiating some flash "event" such as a movie, etc - can be tracked, but only after the flash developers create a flash that can both receive inputs from the Optimost and send outputs to the Optimost.

Here is what needs to be coordinated.

Tracking clicks - if they user click results in a page redirect or new page spawned

The Macromedia website calls this "Building Macromedia Flash Banners with Tracking Capabilities", and can be found at the URL: http://www.adobe.com/resources/richmedia/tracking/designers_guide/

What we want to do is add an Optimost "clickthru" to a flash, and this is done by adding a "clicktag" to a Macromedia Flash "button", and that allows the button's URL to be determined at runtime by this "clicktag" value. So let's say we were testing a flash banner with a single click that tells visitors to buy the product, then to track that click, add a simple action to that button, the action being getURL. The Flash Developer then assigns this "clicktag" to this button, gives it a name, and will inform Optimost what the name is so we can set the flash up correctly to track the link.

So using the flash from Section 1, let's say the Flash developer named it "clickTag", then we would set the flash up as such: 

<object classid="clsid:D27CDB6E-AE6D-11cf-96B8-444553540000" 
	codebase="http://download.macromedia.com/pub/shockwave/cabs/flash/swflash.cab#version=6,0,0,0" width="700" height="300">
	<param name="movie" value="multiVarTest.swf?clickTag=http%3A//by.optimost.com/go/1-8791/a00h00..blah...blah...blah/">
	<param name="quality" value="high">
	<param name="bgcolor" value="#336699">
	<EMBED src="multiVarTest.swf?clickTag=http%3A//by.optimost.com/go/1-8791/a00h00..blah...blah...blah/" menu=false quality=high 
	WIDTH="700" HEIGHT="300" ALIGN="" TYPE="application/x-shockwave-flash" 
	PLUGINSPAGE="http://www.macromedia.com/go/getflashplayer"></EMBED>
</object>

Adding the clickthru as a query string argument in the URL, as in:

multiVarTest.swf?clickTag=http%3A//by.optimost.com/go/1-8791/a00h00..blah...blah...blah/

If you need more than on "clicktag", the Flash Developer can create each one, just making sure that each one has a unique name. Normally, a number is added for each additonal clicktag, as in: clickTag1, clickTag2, clickTag3

Tracking clicks - if they user click results in another section of the flash movie to be shown

In this case, we stay on the present page, so there is no redirection event. Instead, we instruct the Flash developers to create a function call when the event of interest happens, passing Optimost any necessary information in that function call.

http://livedocs.adobe.com/labs/flex3/langref/flash/external/ExternalInterface.html

If customer uses SWFObject Open source library to create the Flash object

SWFObject is a small Javascript file used for embedding Adobe Flash content. You can go to: http://blog.deconcept.com/swfobject/ for more details.

So using this object, the flash above, with our clicktag, would look like this. 

<object classid="clsid:D27CDB6E-AE6D-11cf-96B8-444553540000" 
	codebase="http://download.macromedia.com/pub/shockwave/cabs/flash/swflash.cab#version=6,0,0,0" width="700" height="300">
	<param name="movie" value="multiVarTest.swf">
	<param name="quality" value="high">
	<param name="bgcolor" value="#336699">
	<EMBED src="multiVarTest.swf?clickTag=http%3A//by.optimost.com/go/1-8791/a00h00..blah...blah...blah/" menu=false quality=high
	WIDTH="700" HEIGHT="300" ALIGN="" TYPE="application/x-shockwave-flash" 
	PLUGINSPAGE="http://www.macromedia.com/go/getflashplayer"></EMBED>
</object>
<script type="text/javascript">
	var so = new SWFObject("multiVarTest.swf", "mymovie", "700", "300", "8", "#336699");
	so.addVariable("clickTag", "http://by.optimost.com/go/1-8791/a00h00..blah...blah...blah/"); 
	so.write("flashcontent");
</script>

Flash and Optimost - Implementation concerns for XML implementations

There is a second way to test Flash using Optimost - provide an XML stream directly from Optimost to the Flash to pass the variables values and any pertinent tracking information.

Optimost's XML capabilities

Optimost can serve XML data as well as it serves HTML, CSS, and javascript. The template has to be defined differently, but there are no Optimost-specific limitations to using XML as your served product versus XHTML. When using XML, it is not served directly into the web page - our XML experiments at present have involved feeds directly to Servers which then serve the HTML, or into Flash in a web browser that then modifies its flash movie based on our passed data.

Flash's XML capabilities

Flash also allow us to load data into it using XML. The variables passed in have to be expected by the Flash, which will take appropriate action based on those values. The data passed in usually either images, or "instruction" to tell the movie what to play.

Optimost and XML special considerations - Tracking and Stickiness

The two types of tracking we do are tracking clicks and triggering counters. For tracking clicks, the process is identical to what is described in the section above for modular templates. But for tracking using counters and stickiness, it gets a bit more complicated.

First, since we are only passing XML data, in order to drop our cookies and read cookies, there must be some script code either within the flash or in the container Web page called by the Flash that will read and write any cookies. Since Flash does not support writing a browser cookie directly, we must "call" an external script to write the cookie.

So would pass in the activity tracking and stickiness cookies as name-value pairs the Session Identifies in to the XML, and the name, and then it would call an external function identical to what we use in the Trial code to write out the cookie. For the stickiness, when the Flash calls Optimost, it would have to get the stickiness cookie and pass it back in the XML call.

XML Example

This is an example where I pass 3 images and a clickthru value using XML. Here is the Trial Code URL, in case you want to test and see: http://by.optimost.com/trial/2/p/banneradxmlexample.d31/456/content.xml?size=1

And here is a typical code response 

<OPTIMOST_RESPONSE>
	<OPTIMOST_CONTENT>
		<AREA_A>http://es.optimost.com/es/308/c/2/u/14465_btn_apply_now_brt_green_plain.gif</AREA_A>
		<AREA_B>http://es.optimost.com/es/308/c/3/u/v1.jpg</AREA_B>
		<AREA_C>http://es.optimost.com/es/308/c/1/u/checkmark-green.gif</AREA_C>
		<CLICK_TRACKING>http://by.optimost.com/go/2-3282/a0fb0hw21j245ei27k07n7904/</CLICK_TRACKING>
	</OPTIMOST_CONTENT>
</OPTIMOST_RESPONSE>

XML IE6 Bug - Please Note

In an implementation, the Flash will be making a cross-domain call. That's fine, Optimost has a file that tell the Flash we are an "OK" domain. The IE6 issue has to do with GZIP compression. At the subject level, under Change subject options, you need to select never and all will be fine.
