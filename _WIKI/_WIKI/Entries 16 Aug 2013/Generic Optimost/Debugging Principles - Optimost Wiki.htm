Debugging

I believe that this is where the Technical Consulting team needs to improve the most as a group. This is an area that is as much art as science, and we all need to have sound methodology that we follow in order to uncover what the issues we experiencing are, and thus how to fix them.

First Principles

In Grad school, I had a Professor who was an MIT grad and had been taught by Claude Shannon (and if you don't know who Claude Shannon is, shame on you, stop playing World of Warcraft and Google him). He said Shannon was a real character, he use to ride his unicycle around campus, and was not a very good teacher, but he did say one thing all the time, and that was "always go back to First Principles". And that is my advice to you.

First Principles are the base assumptions we know to be true about our subject matter. If you violate a first principles, that means what you are doing will not work. For example, rule #1 for Civil Engineering and Plumbing is:

  1. Water runs downhill

So that means if you downstair's neighbor is distracted talking on the phone and her sink overflows, I wouldn't sweat it. If this happens to your upstair neighbor, you might (as I recently did) notice water raining down on your refrigerator.

Four Rules for TCs

Based on the concept of First Principles, here is my list of the four things I consider when debugging a Template. You may notice that these are the things I do when you ask me for help:

  1. Make sure the page is integrated in the clients environment (CSI code deployed) properly.
    • The Page code is there and the CORRECT trial code call is being made - check the Trial code URL and compare to the expected URL
    • Make sure the Trial code call is returning the content you expect - look at the content, check the names of modules, cookies, etc
    • Make sure doing sticky=99999 give you the "default content" when our Trail code returns "failover"
    • Make sure the CSI code has been deployed correctly structurally - especially an issue if we are doing rendered content, not DOM manipulation.
    • Familiarize yourself with the expected content.js full URL, see what the Page code pulls out attaches to the URL, you should know this by heart.
  2. If there is an error on a properly integrated page, it is most likely something I have overlooked
    • This is a nice way of saying I did something stupid
    • Please assume its your fault first instead of blaming someone or something else right off
  3. When you "find the solution", make sure to test it properly before declaring victory
    • Clients do not like it when we say we fixed something and it is in fact not fixed
    • Doing this can make us look careless/incompetent, arrogant, and disinterested
  4. Once the solution is verified by testing, make sure you learn something from what you did
    • This means you understand what the problem was, and why your solution fixed it
    • Make sure you don't repeat the same error again.

Digging into Rule #2-3

Ok, so there is an error in your code. How do we figure out what it is? Remember the following rules:

  • When debugging, make sure with absolutely, 100%, rock-solid certainty that the code is doing what you think its doing. This is the #1 issue you need to remember. So many of our "bugs" have to do with thinking our code is doing one thing, when in fact it is doing something else. So what I do is put alerts all over the place to see when the code stops running. Here is a good example of something you could waste an entire day on and get nowhere.

So one of us was running a test that was in QA, and it had no javascript errors, bat was not working as expected. We had the creative # showing up in the Title bar at top, which told us that the Trial code was being executed, but since the content was not being changed, that meant the module code was not working as expected. How to figure it out? Here is the module call: 

<script language="text/javascript">
optimost.displayModule("Body");
</script>

So the TC came to me after hours of looking at this, having "checked everything", and asked for help. I asked "did you put an alert in the first line of the module to see if it gets executed?" They said yes, and it was not triggering. So next I asked "did you test to see if the module call gets executed?" They said no, they can't really test that. I said "not true", so, in Firebug, we clicked the "console" tab, and on the right, put in optimost.displayModule("Body"); and hit run, and suddenly, the content appeared. Hmmmmm. This means that somehow the module was not being executed as expected, but the module call was defined and could be executed.

I asked if they had taked the source and put it on our servers to test it, and they said no, figuring it was in the client's QA and putting it on our servers is superflous. So we did - put the CSI up there with a base statement - and then put an alert in front of the module call, as in: 

<script language="text/javascript">
alert('hi');
optimost.displayModule("Body");
</script>

and the alert did not trigger..... what?!? Then I looked more closely at the script tag.....<script language="text/javascript"> is an invalid opening tag, it should be:<script type="text/javascript">

So we figured out in 10 minutes what they had wasted hours debugging by making no assumptions and just

  • So in order to be absolutely, 100%, rock-solid certain things are working, put alerts in to see where your code fails.
