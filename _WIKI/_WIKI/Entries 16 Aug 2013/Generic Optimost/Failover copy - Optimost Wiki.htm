Failover copy

An ignored feature of Optimost in the present incantation of our Templates building process. When we first started as a company, this was an absolutely essential element of our offering, and without it we would not have been able to sell our product. So here is the past, present, and other meanderings.

For more information on Failover copy, please see Optimost Creative Serving network

As you can see, there is no Page Code, no Hidden Spans around the content, no module calls. We have a script call that has an Optimost URL and only sends the query string parameters and the referrer back to our servers. And the code we are testing is set below our call in a noscript block. This meant that if the browser did not run script, which was a possibility back then, our code would not run and the default page would display.....

Nice, but what about when our Servers don't respond, what happens then?

Failover Served when....

Failover had to be implemented becasue, if our servers are down, then you would get no content and a blank test area — very, very bad. We would not have been able to sell Optimost if this was the result of our Servers going down - customers need to make sure our product is reliable when bad things happen.

So Akamai offers a product to accoutn for this scenario called Failover copy. All Optimost content is served from multiple, geographically distributed, fully-redundant data centers, and each request is optimally routed through the Akamai network to our Servers. If Akamai finds that they cannot get to our Servers, then they serve the Failover copy, a "control" copy of the page (which the Optimost TC will save at the Subject level).

So in the past, this was very important to avoid a blank page. But as our product has evolved, it is less important, and we will cover that shortly.

How to emulate serving of Failover

In the past, they way we emulate it was to add a query string paramter

"?opfailover=404"

to the URL of the page being tested. Today we do the same thing by using a sticky value that is not in the test, as in:

"?sticky=99999"


Our Product today

Today, for our module test - which represent 95%+ of our tests - our implementatation wraps the default content in a hidden span, so that if our servers are down, we in fact want an empty page to be served so that the spans are not hidden and the default content is shown. So for the overwhelming majority of test, the failover copy should be blank.

So, on this page, the javascript call rendering "nothing" results in the default page showing. This means failover should serve no content, which is what will happen if you put nothing into "Change Failover Copy" at the Subject level. If you click on that selection, you will notice there will be one tab for each module, and a tab for the setup block.

With an empty Failover copy, the output from content.js for a module Template will look something like this: 

document.write('<style type="text/css">' +
'.opDefaultContent_nameOfSubject' +
'{display:none;}</style>');

which any of you who have added ?sticky=99999 to the URL will have seen as the output of content.js.

So do I ever need Failover Copy?

Modular Templates

On a modular Template, the answer is "not often". Content wise, if we use the normal setup - setup each module to show the default content when we render nothing - then we need to add nothing. But in an instance where the client is expecting something from us, then we need to add something to failover. Potential scenarios include:

  1. You are doing a dreaded A/B test, and it turns out the styles sheets are different enough that you have to remove the styles from the default page and insert them individually for each version as part of the variable value (either in the head via the setup block or in the body via a module). In this case, the removed style sheet(s) need to be added to the appropriate Failover Copy module or setup block
  2. They are expection tracking codes to be set as javascript variables by Optimost that they will pick up below our code. Then we will need to add something to set those javascript variables to let them know we are not rendering content.
  3. It's an AJAX implementation and the customer has added function calls - hooks - to their code to allow us to modify the content we are testing after their AJAX call is finished making its changes. We would just need to add the function definitions, and make them empty functions, as in: function hookToOptimost(){}

And there may be other scenarios I have not thought of. But if one of these or another comes up, also remember that now that the failover copy is not empty, Optimost detects there is content in the Template and adds code to content.js that will HIDE the module's SPAN tags - as in:

<style>#opmodule_body{display:none}</style>

So, you will have to add code to the Failover to turn on the content, as in:

<style>#opmodule_body{display:inline}</style>

Other Types of Templates

For other Templates, the question is ¿Have we setup the test to show default content when our code does not render?. If the answer is yes we have, then failover is not necessary. If the answer is no, then the we must setup failover. So briefly, the other Template types:

  1. Email Testing—MUST CREATE FAILOVER COPY. This is generally a Server to Server process, and since you can't run javascript, we can't set the pages up to show default content.
  2. An XML implementation—MUST CREATE FAILOVER COPY. This could be a Server to Server process, or between Optimost Servers and a Flash, but whichever, and since you can't run javascript, we can't set the pages up to show default content.
  3. A Test that test the entire Web Page—MUST CREATE FAILOVER COPY. A full Web Page test means we are not serving Javascript code that then renders the content, but the actual HTML page is output, so again, no default content, we must have Failover.

And there may be other scenarios you encounter. Think it through, ask others what they think, and determine if you need Failover Copy.

Make sure to test Failover when the test is in QA

Don't forget this for all your templates. ALL. Yes, should be OK on proper modular implementation, but we need to make sure we are covered if all our servers are down - if Akamai is down, well the Internet is in trouble, we can't account for that.

SO remember to invoke failover:

"?opfailover=404"

to the URL of the page being tested. or we can also do:

"?sticky=99999"
