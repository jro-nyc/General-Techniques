INTRODUCTION

Here is a FAQ geared towards "inquiries made by potential customers to the Sales Engineers". If you cn think of other questions that should be here, please contact me - Jim Romero.

QUESTIONS

Explain what Optimost does?

A broad definition is that Optimost uses the Internet to serve ASCII text using secure or non-secure web connections. This can include:

  1. HTML/CSS and/or javascript served to a web browser using a script include, and that can be:
    1. A Multivariable test (MVT) that test a number of page elements and their interactions.
    2. A/B/n tests that test different versions of a page
    3. Serving ads embedded in Web pages.
  2. Serving XML to:
    1. Flash embedded in a web browser
    2. A customer Server that uses the XML to generate a web page/email on the back end
  3. Hosting Landing pages for customers
  4. Serving ASCII text in some custom application

Most of our experiments – over 90% – are type 1.1 or 1.2. In these experiments, there is no interaction between the Optimost servers and the Customer’s servers. The interaction is between the Optimost Servers and the visitor’s Web browser, and once the browser loads the content, the changes to the page are made in the browser.

Explain how Optimost will interact with our back-end systems?

This is a relevant question only if you are using XML to receive content from Optimost. Otherwise, there is no interaction between Optimost and the customer’s back-end, other than the Optimost code being place within the HTML of the page(s) to be tested as you would any standard include. If you are using XML approach, you will need to provide the data set you are expecting Optimost to pass you the content for the experiment.

Explain how Optimost will integrate with our back-end systems?

As explained in the last point, Optimost works in the browser. So, Optimost will provide a “Client-side Integration guide” that shows where the code needs to be in the final HTML. This will consist of:

  1. Page Code and Trial Code in the head– javascript code that makes the call to our servers
  2. Module Code below the test area – a javascript function to make changes to the page under test

How large are the libraries (specifically JS)?

I guess in this context our Page Code is the library – probably not a real library in the sense most customers would mean it (large, complex libraries like JQuery, Prototype, YUI) but it does have pre-coded functions/methods used to more easily implement our tests, so let say yes - and the Page Code/Trial Code and Module code is under 3k, and is normally added to the page - we call this the CSI guide.

Where is the JS library hosted (local or remote)?

The CSI guide is usually on the page, but can be stored remotely and place on page as includes. We are fine with the Page Code being stored remotely since its universal for all tests, but care must be taken on Trial Code and modules since they are test-specific.

How do tests change the page size/weight?

  1. Not A/B test – If we are doing DOM changes in the experiment, the content is usually under 15k.
  2. A/B test – dependent on the size of the HTML for the challenger versions, plus a few k for the rest of the overhead

How is our Page's performance impacted by the test?

Page should work as normal. As for "load time", the content loads quickly - get numbers from Huge - there is a possible appearance issue. We normally hide the content when the Trial code call is made, make changes and make it visible when the module is called. If there is a lot of content being loaded in the body of the page above our modules, we have to wait for that to load before we can display the content, and instead of the page appearing as a waterfall, it can suddenly appear after a delay. This has less to with our load times and more to do with where they place the module and/or how heavy the content in the body is.

How abstracted is code for test from production code?

If they mean the "QA" versus "Live/Production" environments, there are two ways to do it.

  1. We provide code for a "live" trial only, and we use Personas to separate the Test from Production content - usually a query string parameter, a cookie value, or, in the case of them actually having a separate test environment, the domain information. This means they can load the same code in test or production, and then use Personas to “test” the QA version. We do this 99% of the time since it’s convenient for the customer, and it means we can QA in whatever environment they choose to.
  2. We can provide code for both "live" trial and a "QA" trial, and they will place it as appropriate. This is fine with us, but the biggest disadvantage is that because of the way our Trials are architected, if they wanted to go from Trial 1 to Trial 2 with the QA trial still there, there may need to be new Trial code deployed. Feel free to talk to Lance about that one :)

Do Optimost tests pass standard 3rd party JS standards?

  1. For cookies, if we drop 3rd party cookies, they have a P3P privacy policy, the is no Personally Identifiable Information (PII) in any cookie we drop.
  2. Our "library" - the Page Code - defines its own object and functions, and does not interfere with other libraries or the Browser's native commands. And emphasis we don't consider the Page Code to be a library in the broad, general sense that most IT folks speak of libraries, it’s a library in the strictest “pre-coded functions/methods used to more easily code” sense.

Does your tool deliver content to the page or does it require DNS routing?

The client continues to host the page – they deploy our CSI guide on the page, and when the Web page loads, it calls our servers, and our content makes the appropriate changes. No need to do any DNS routing since they will continue to host the page.

Does your solution affect organic search results? Please describe, in detail, how your solution avoids this challenge.

When our CSI guide is deployed, it does not remove any HTML/CSS/Javascript content from the clients page, it adds our Page Code, Trial Code, and Module Code, which are blocks of javascript. The search engine spiders sees any page as a text-based, Javascript disabled entity, so the page with the Optimost CSI guide will look exactly as it did before the guide was added.

Is there a concept of ‘default’ content to ensure a consistent customer experience in the event of non-response or suspended testing?

Yes, in fact that is the intent of our deployment architecture. When the CSI guide is deployed, our server are configured to return no content for that experiment, which result in the default content appearing. And if our servers are down, Akamai store a blank “failover” copy to serve the same purpose.

Can your solution support "off-site" testing, such as SEM ads, display ads, images etc?

Yes we can, either as Standard deployment (Using a CSI guide) or as standalone Flash application (using our XML-based deployment). We recommend the XML-based Flash deployments as the best way to test off-site advertisements because of the ease of integration – you simply program the Flash to call our servers to obtain the MVT data, no need to make any changes to the off-site page.

Does any part of your solution require us to install software on our servers?

No

Does your solution support different operating systems and development languages? (e.g. LAMP, .Net, JAVA. PHP)?

Optimost is a front-end, browser solution, not a back-end solution. We work in any browser because browsers all understand HTML, CSS, and Javascript. (And in the case an older browser that does not understand Javascript, or in a browser where the user has turned it off, we do nothing and the default content will appear). The back-end code is only needed to place the CSI guide on the Web Page when a request comes in from a visitor, and that can be done by any back-end.

How will the solution be implemented on the software side?

It is a front-end solution, so we will use javascript to implement all changes. This could entail adding new HTML, modifying or adding new CSS, or making changes to the DOM. For most Standard implementations, the changes to the page are done by making changes to the DOM or CSS, and a small minority may have actual HTML added to the page (not via the DOM).

Does your tool support the use of Akamai Caching, or any Caching?

There are two consideration here:

  1. Since each visit to our experiment – what we call a “Page View” – has unique tracking information that must be generated by our servers, the pages can not be cached.
  2. Also, since we use the same URL to make a request to our servers, and that request could serve any number of different creatives for the test – ie, the decision of which creative to serve is made by our servers – the pages cannot be cached.

Can we run multiple experiments on one page?

Optimost can run multiple experiments on a single page, but we only recommend it when it is prudent to do so – ie when it needs to be separate experiments. So:

  • If you want to run a universal Header test across your entire website, and concurrently run tests on individual pages, we would recommend those separate tests. The two experiment are truly separate and can be analyzed independently. Here we would say YES.
  • If you want to test a number of content areas on a page – say you are testing:
  1. The Left navigation
  2. The headlines and sub-headlines in the main content
  3. The “call to action” buttons on the page
  4. The layout of some of the elements

this should be one experiment testing all the elements. To break it up into multiple experiments makes the analysis of the results much more difficult. Here we would say NO.
