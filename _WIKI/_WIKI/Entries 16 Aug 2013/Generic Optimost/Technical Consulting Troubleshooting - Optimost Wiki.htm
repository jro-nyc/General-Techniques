Scenarios and Issues

The purpose of this list is to bring up situations we deal with and how to analyze them. Every TC should be able to look at each question and answer it.

A/B/n test vs Multivarible tests

  1. Why are A/B/n test often harder than Multivariable test?
    • Most of our tests are done by the TC making the necessary changes by modifying the DOM. On a Multivariable test, we are making incremental changes to the control page, so modifying the DOM is the logical solution. In an A/B test, usually the A/B/n versions are so different that we cannot make incremental changes to the structure of the control page to produce the other pages. Thus we have to render the different versions as HTML, and the limitations of rendering the HTML can cause us problems.
    • We tell customers that we need assets and "working" pages for A/B/n tests. There is often a disconnect here since we will normally have a working A version, but often the other versions are anything from wireframes to standalone HTML implementation (ie visibly correct but not a working page) to a working page in a different location that is not "compatible" with the A page.
      • Wireframes and screenshots are unacceptable, we need working HTML
      • Standalone HTML pages are also unacceptable, often the functionality is not compatible with the A page - different form input names or form functionality, links not functional, etc.
      • A "working" page that lives on different servers with different properties. Again, the page needs to work alongside the A page. It may be compatible or not, but the customer needs to understand that working pages means functioning like real pages on their site.
  2. What constitutes difficult A/B tests
    • If an A/B/n test has any dynamic content, it is more difficult to handle. We need to remove the dynamic content from the page, store it, and then re-render it on the other versions. This is not easy to do for a number of reasons, and if the page also has business rules, that further complicates the process.
      • Ajax can be especially difficult to handle if we do not get working B/n pages.
      • Remember, ad-serving is dynamic content from an Optimost perspective. This complicates implementation.
    • Test with different A/B/n versions having different style sheet definitions. This can complicate implementation since we cannot "hide" style sheet differences the way we can "hide" DOM differences. This complicates the creation of the CSI guide and its implementation.
    • Trying to test different Form pages in an A/B/n test.
      • All the Form pages must be able to "work" based on the A version since that is the one hosted by their servers during the test.
      • Need to communicate this to the customer so they configure the pages correctly when they hand us the assets for the test.
      • We deal with this issue - lack of understanding of how we work - from the customer on probably every single A/B/n form test since its usually a marketing guy, not a tech guy "creating" the test.
    • Tests with versions containing different DOCTYPES
      • May be impossible to implement since we cannot work around the DOCTYPE limitation.
      • Manifests itself as B/n pages looking "broken" layout-wise
      • You may need to define a head and body module - covariables, make experiment standard MVT - and separate the content
      • REMEMBER - must add default head style content to failover AND tell customer to remove style sheets from head
    • Test with different A/B/n versions having different javascript definitions
      • Manifests itself as B/n pages triggering JS errors and/or having broken functionality
      • You may need to define a head and body module - covariables, make experiment standard MVT - and separate the content
      • REMEMBER - must add default head js content to failover AND tell customer to remove js definitions from head

Failover Copy

  1. What is it?
    • A single copy of the content to be tested that is saved by Akamai in case they (ie the visitor) cannot get to the Optimost servers.
  2. How it is implemented?
    • At the Subject level, on left menu bar >> Change failover copy
    • You, the TC, would add the failover content to the appropriate modules and setup block, IF NECESSARY
  3. How to test it?
    • On a page in QA, add as a query string paramater "opfailover=404"
      • Alternatively, "sticky=99999" will do the same thing
  4. IF NECESSARY - When do I need it?
    • On modular and snippet templates, we normally hide the default content in a "hidden" area which is hidden by the Optimost content, then modified as need by by our content. In this scenario, our content responding with nothing results in the default content working as if we were not there. So for this scenario, assuming customer pageis not expecting some special Optimost behavior, leave failover empty.
    • XML templates must have some failover, and also must coordinate how to handle clickthrus in failover scenario with customer.
  5. Tricky stuff - have you covered all the different potential scenarios?
    • There is a single copy of failover for the entire subject. So....
      • ie. A multiple Persona template, will Failover work for all of the Personas?
    • If we are testing both content in the body and the head, how is this handled...
      • Content in the body can be hidden by either wrapping it in our standard span tags OR, in the instance where the test are has a unique class or ID, adding a style in the Trial code setup block to set display:none.
      • But CSS/Javascript content (in the head or body) cannot be "hidden" in the same way that HTML content in the body can since it does not display. We need to actually remove the tested content - style sheets and/or javascript functions - and add it as the control value for that variable (which will likely be a covariable, say {{$AreaA.head}} and {{$AreaA.body}}). And, likewise, the control value for this variable will need to be in the failover.
    • What if we have an AJAX implementation and the Customer has provided "hooks" - function calls in their AJAX code to Optimost so we can make our changes after they modify the content - how do we handle this?
      • Make sure one of the modules or setup block have the function defined and does nothing, as in myAjaxFunc(){}
    • What if the customer is expecting Optimost to set something for them to pick up - say a tracking pixel with the creative and value information, or setting META tag information, or js variables for their analytics, etc?
      • Coordinate with the customer this contingency, and again, set something in the Failover to indicate their default content is being displayed. This could be nothing, but account for it.

First vs Third Party cookies

  1. Why use one versus the other
    • If page-under-test and entire visitor funnel are in the same domain, normally First-party is the way to go.
  2. The advantages and disadvanatages:FIRST-PARTY
    • [+]First-party cookies almost never get blocked by users, even paranoid ones.
    • [+]First-party cookies can be accessesed and modified at template level, if you need to do this at run-time (as in adding attributes and values as cookies starting with "op" to be captured later by counters).
    • [-]First-party cookies, since they are added to existing customer cookies, can go over the browser's cookie limit if we add too many.
    • [-]First-party cookie modification by third-parties (Optimost) may be an issue for some paranoid customers.
    • [-]First-party cookies need custom javascript code to read them.
    • [-]First-party cookies cannot be used directly in Flash XML templates, need customer assistance to implement.
  3. The advantages and disadvanatages:THIRD-PARTY
    • [+]Third-party cookies will work across domains.
    • [+]Third-party cookies are Optimost cookies, so we don't need to worry about the cookie limit or customer opposition.
    • [+]Third-party cookies - Browser automatically sends back all Cookies in external call domain, so trial code and counter code calls return all Optimost cookies automatically.
    • [+]Third-party cookies will work with Flash XML implementations automatically.
    • [-]Third-party cookies are more likely to be blocked by not just paranoid users, but also many web security add-ons.
    • [-]Third-party cookies can NOT be accessesed and modified at template level - so we cannot add attributes and values at run-time.
  4. Although it might appear at first glance that third-party cookies are better than first-party cookies based on the pluses and minuses, the potential for blocking of third-party cookies makes the less desirable to use.


Refreshing creatives - Subjects, Trials, Personas, Waves, Templates, Creatives, what they all mean to the TC

  1. What is a Creative?
    • Creatives are the content for an experiment, the different "version" of the page that will be tested. Each creative is a single permutation of a Template. Creatives are saved to our Servers when a Wave is activated, then when that wave is set as the current Wave for a Persona, it will be served for visitors that meet that Persona's criteria.
  2. What is a Template?
    • A Template contain all the content that will be optimized. A TCs main job is to create Templates based on the instructions provided by the STP (Subject Test Plan). Each Creative is a permutaions of a Template.
  3. What is a Wave?
    • A Wave is a test of a specified set of creatives run against a persona of visitors within a trial. Noramlly, the wave contains creatives form only a single template, but can have creatives from different Templates.
  4. What is a Persona?
    • A persona is a group of visitors with common traits participating in a trial. The "traits" are determined by the Persona criteria - the set of conditions that define the group with common traits.
  5. What is a Trial?
    • A trial is a test of a subject against a group of visitors (referred to as an audience). Initially the audience is tested as a whole. You can divide your audience into smaller personas if you want to optimize creatives for each particular persona
  6. What is a Subject?
    • A subject represents the individual Web page or content being tested. A registration page, a landing page, a registration page, or a pop-under advertisement can all be subjects.
  7. OK, so I have a Wave in Persona, and them I make a change to the Template that the Creatives are based on. What happens?
    • As mentioned in the "What is a Creative?" section, creatives are saved when you activate that wave. This means that any subsequent change WILL NOT be automatically propogated to the creatives. In order to propogate the change, you must REFRESH CREATIVES.
  8. How do I Refresh Creatives?
    • This can be done in two ways:
      • At the Template level - when a wave is created from a Template, the left side menu will now have Refresh creatives, and clicking on it will Refresh all creatives for this Template. Most of the time, we Refresh at the Template level.
      • At the Wave level - when you either have multiple Personas with Waves from the same template, OR a Wave with Creatives from multiple Templates, you may want to Refresh at the Wave level. In the former case, it allows you to leave one Persona untouched while you test the other, which is great for doing QA. The latter scenario means you can just do a single Refresh Creatives, versus having to go into each individual Template in the wave and Refreshing Creatives
  9. What changes require a Refresh?
    • Basically, anything you change within the Subject or Portfolio except for changes to Personas or to Failover copy. Here are few examples:
      • Changes to the Template - yes, obviously
      • Any changes to stickiness - Any changes to the Change stickiness menu item at the subject level.
      • Any changes to activity tracking - Any changes to the Change activity tracking menu item at the subject level.
      • Any changes page view attributes - Any changes to the Set page view attributes menu item at the subject level.
      • Any changes to the internal domains or clickthru domains - Any changes to the Set clickthru domains menu item at the subject level or the Set internal domains at the Portfolio level.

Forms - The "Gotchas" I should look for before I start and during implementation

  1. Is the form a POST or a GET?
    1. You must know the difference between the two
      1. GET sends data in the query string of the action URL
      2. POST sends the data in the HTTP Headers
    2. If POST, remember to edit Manually-created clickthrus to be a POST
      1. Manually created clickthrus will initially be GET clicks
  2. Are there any inputs named action?
    1. May interfere with changing Action attribute on Form Post.
  3. Are there any inputs named submit?
    1. Will interfere with trying to track the click
      1. form.submit() cannot work
      2. Not recommened behavior - naming inputs "submit" - as per W3C spec, but clients often do it unknowingly
  4. What is the enctype for the form
    1. default - no enctype tag, which is normally the case - is application/x-www-form-urlencoded, which we can track
    2. If you see enctype="multipart/form-data", we cannot track this click.
  5. Does the Form have an onsubmit event that changes the action URL?
    1. Look for this if your clickthru is somehow "disappearing" after you add it
  6. Did you add an onsubmit event to track clicks that is somehow not being executed?
    1. Is the form submited by a function that calls formName.submit()?
      1. If so, remember that calling submit() submits the form WITHOUT executing the onsubmit event
      2. The work around is to add the call to the onsubmit event just before the submit call, as in:
        • formName.onsubmit();
        • formName.submit();

Tracking 101:Why are my clickthrus not working?

  1. Understand how clickthru creation works from an Optimost Template perspective
  2. For templates where we do "rendered content" - ie, non-DOM method templates, generally A/B tests - the following applies:
    1. When going through the template wizard, anchor tags, Form tags, Area tags, and Map tags are given clickthrus on their hrefs or actions.
    2. A "spoofstatus" piece of code is also added
      1. onmouseover and onmouseout events to spoof window.status
      2. Newer browsers ignore this, which is why we are going to onclick events.
  3. If you are using the DOM method:
    1. Obviously, there will NOT be any hrefs or actions for the Template Wizard to add clickthrus
    2. You will have to add then manually to the template - {{$clickthru.urlTagName.url}}
    3. After creating them in the Wizard, edit the clickthru to add a the request argument for this dynamic click
      1. Always select "URL will always be specified by request argument"
      2. Select "URL can be overridden by request argument" may result in TC not realizing their clickthru code is broken because clickthru appears to be working
  4. If clicks are dynamic and you get a "page not found" or "Error 404 not found", make sure to check the Internal Domains!!
    1. If you are redirecting to some URL, need to have allowed its domains in:
      1. At the Portfolio level, the Set internal domains menu item under Other Tasks, OR
      2. at the Subject level, set the domains in the Set clickthru domains menu item under Other Tasks.
    2. This is for security reasons
      1. Without this, some rogue entity - PC talk for a bad guy - could take our Optimost URL and redirect people to bad websites
      2. As they say, "Fool me once, shame on you, fool me twice, shame on me"

Tracking 101:Why is my counter not triggering/working?

Check one of the following:

  1. Is the counter on the page we want to track?
    1. Please, PLEASE verify this first. Someone must make sure the customer put the counter on the appropriate page.
    2. If we ASSUME it is there, we are setting ourselves up for failure.
  2. Is the counter in question associated with the correct subject?
    1. If the counter is not associated with the subject, the action from that counter will be ignored.
    2. Go to the Change activity tracking link in the Subject level and in the Counter tab, select the counter.
  3. Is the counter on a page in a different domain than the test page?
    1. If pages are switching domain, you will need to use cross-domain cookies.
    2. Go to the Change activity tracking link in the Subject level and in the Cookie tab, select cross-domain for the cookie domain.
  4. Are the reports up to date?
    1. Sometimes there is a lag in our reporting.
      1. Even though the reports say "* Today's metrics include activity through 10:40 AM EST" at 10:44 EST, the numbers may still not be up to that time.
      2. Large volumes of traffic result in delay of data migrating to our Database
    2. To see latest times, set up a Daily report with 5 minute increments. To do that:
      1. Go to the Wave of interest. On the left side, Under the header that says Reports, click on the view link and select daily
      2. Now you are in the wave, on the left click Customize..., and under the General tab, in the Time Period select, choose Daily in 5 minute intervals and click OK.
      3. Scroll to the bottom of the report and see the last 5 minute interval with an action from that counter, or any activity in general.

Response Time Issue

  1. For Page Loads - What is the issue
    1. Is it an Optimost issue, and if so, what is the issue
    2. Is it load time from our servers
    3. Or is it the time to when our module makes the content visible
    4. If the issue is the load time of their content between our Trail code call - turns display off- and the module call - turns display on - what can Optimost and the Customer do about that?
  2. For clickthrus
    1. Is it a delay in response from us or from their servers
    2. So is the problem the blank 302 redirect page beacuse their QA servers are slow, and if so, what can we do about it.
  3. Present Statistics - as of September 2008
    1. Average Load time: 35 to 65 ms for the last 6 months (This number comes from Akamai)
    2. From a "monitoring device" they have hooked to the Internet
      1. This device probably has a T1 to the Internet that never gets bogged down
      2. So your mileage may vary



Converting CSS style definitions to DOM style object equivalents

  1. Formula is simple:
    1. Remove the dash, capitalize the first letter after the dash
    2. So style="text-align:center" in CSS becomes style.textAlign="center" in javascript
  2. So why am I having problems applying this rule to the "float" property?
    1. Turns out that float is a reserved word in many languages, including javascript :(
    2. So you must use a different identification for javascript
      1. style.styleFloat for IE
      2. style.cssFloat for standards-compliant browsers


Optimost <SPAN> tags around modules

  1. Why do we do it?
    1. To delineate the area we are testing
    2. So the system can automatically hide the area - Optimost sets the ID's style to display:none
  2. When do we absolutely need them?
    1. If you are not going to find another way to delineate the test area AND hide it yourself, then you need them.
  3. When do we not have to use them?
    1. If the page has the test area encapsulated in an HTML element with a unique ID, we can theoretically omit the tags and do the hiding work ourselves.
    2. Must write the code intelligently to take all factors into account
  4. Of course, you must still use the module call code, no way around that :)

Set internal domains at the Portfolio level and Set clickthru domains at the subject level

  1. What is each for?
    1. To allow us to specify where our redirects can go, and thus, where they 'cannot go - everywhere else.
    2. This give us more control and better security in where our clickthrus redirect
  2. When do we need to worry about it?
    1. Without this, some rogue entity - PC talk for a bad guy - could take our Optimost URL and redirect people to bad websites
    2. As they say, "Fool me once, shame on you, fool me twice, shame on me"
  3. If we need to worry about it, what do we need to do?
    1. Make sure you find out from the CRM what domains the customer's page will be redirecting to.
    2. Unless there is some really unique situation, you should NOT "allow any domain" at the subject level.


The Identity Attribute

  1. What is it?
    1. An extra "attribute" when the attribute is in fact unique, or if there are hundreds of attributes the customer wants to collect
      1. Our regular attributes can only collect up to 100 "attribute values" and count how many of each value you got
      2. After the 100th, any new values will be ignored
    2. Cannot be seen in the Agency reports, can only be access by downloading the reports and processing them offline.
  2. When should we use it?
    1. If the customer wants to collect unique information, usually for order correlation
      1. Thing like Social Security numbers, Customer Order IDs, Session Identifiers are unique

Setting onmouseover and onmouseout for rollover effect

  1. If you say onmouseover="this.src='/images/newBlah.gif';" and onmouseoout="this.src='/images/origBlah.gif';"
  2. Remember to end with "return true;", as in onmouseover="this.src='/images/newBlah.gif';return true;" and onmouseoout="this.src='/images/origBlah.gif';return true;"

Named Anchor tags and hashes - an IE6 issue and its fix

  1. Let's assume you have <a href="#wine">Get Wine</a> and later in the page you have <a name="wine" >Wine List</a>
  2. If we are rendering this code out as an A/B test, there is a chance that although the "original page" did not reload when you clicked on the "Get Wine" link, it may reload in IE 6.
    1. Known IE 6 and below issues, behavior not consistent.
  3. As a workaround, find out if the browser is IE, and use a proprietary IE function to move focus to that tag, as: <a href="#wine" onclick="if(opIsIE!=-1){document.anchors.item('wine').scrollIntoView();return false;}">Get Wine</a>

IE "access denied", but not with Firefox - document.domain changed by client

  1. You might notice the page "not working" in IE but fine in Firefox
    1. You will get an "access denied" error in IE
    2. The Trial code call may not be made
  2. Modifying document.domain results in an error being flagged when the optimost.SLD() function is called
    1. It tries to get the value of document.domain, and for some reason you get "access denied"
    2. Fails and thus anarchy ensues.... ok, maybe the trial code call fails.
