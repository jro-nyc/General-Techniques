"Broken" clicks

From time to time, you may be testing an Optimost clickthru, and the clickthru will result in a page not found, 404 error. As a TC or QA person, you need to figure out EXACTLY why this is happening - "there's something wrong" is not enough - so here we will go over clickthrus interact with the Optimost servers, Customer servers, and possible reasons why you would get a 404.

HTTP codes

First, you must understand that a 404 is an HTTP error. After any HTTP request, there will be a response with an HTTP code. The following page has a quick summary of HTTP codes, please review this page briefly:

http://www.modemhelp.net/httperrors/httperrors.shtml

As you can see, the codes are broken up as follows

Value Description
1xx Information Codes
2xx Success Codes
3xx Redirection Codes
4xx Client Error Codes
5xx Server Error Codes

In fact, every tracked clickthru that is a GET uses a special code - 302 - to redirect the page after we track it. And most requests result in a 200, which is the base success code.

HTTP 404 - Page not found

A page with a 404 falls into the Client Error codes category ,and the error means:

"The document that has been requested either no longer exists, or has never existed on the server."

Anyone who has used the internet regularly has seen this error - all too-common for search results pages - and we just take it that the page is no longer there. Often, you will not get an HTTP error, as some Servers, when getting a request for a non-existing page, will either redirect you to another page that does work, or will give you a custom version of the 404 error that is a message from them, not just the generic HTTP 404 error message.

In January of 2008 Optimost features we spoke about last month was adding Clickthru Dynamic Target Security. Please read the attached pdf for more details as a refresher. In any event, if you fail to add a domain that you are trying to access to the list, you will get an 404. This includes their QA environment – you may need to add these, too.


So for example, let’s use the link below. You will see that if I do the following and put this as the browser URL:

http://by.optimost.com/post/335-1732/a00h00y0682424h0kx15982424h0kl2r8be5f/?oplink=https%3A//www.jaimerocks.com/subscription_offer.asp

And hit enter, I get a 404 – jaimerocks.com is not on our internal domains list. However, trails.com is on the list, so if I say:


http://by.optimost.com/post/335-1732/a00h00y0682424h0kx15982424h0kl2r8be5f/?oplink=https%3A//www.trails.com/subscription_offer.asp


everyone is happy.

This is done at the Campaign level, under Other Tasks, Set Internal Domains. When you set it, in this case, just say trials.com, no need to say www.trials.com and narrow it unnecessarily.


Be on the lookout for this error, please, and know how to fix it.
