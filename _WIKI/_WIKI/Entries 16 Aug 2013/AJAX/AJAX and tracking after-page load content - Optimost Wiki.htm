AJAX and tracking after page load content

More websites are using AJAX to efficiently pass data from servers to web pages without a page refresh. One of the more popular applications of this is auto-populated search bars. This entry goes over the two options for adding tracking to those elements that are loaded via AJAX.

As you can see, there are a lot of nodes added and the content has changed considerably. How can I not track user actions on these new nodes?

Approach #1 - "hook" into the AJAX code

The first approach is to get a "hook" into the AJAX code to add tracking code. Somewhere on the page, there is AJAX code that handles the transaction between the browser and the server, and when the content comes back from the server, right after the customer's code makes the changes to the DOM, the Optimost code is called to make its tracking addition. You can reference the following WIKI entries for more on that approach:

  • AJAX overview
  • ASP.NET_AJAX_and_Optimost

The biggest disadvantage of this approach is that you need to have way to hook into the AJAX code. The viability of doing this can be categorized as follows:

  1. In the .NET example in the wiki, Microsoft has built in an event that gets run when AJAX updates are made. So we can use this functionality to "hook" ourselves in, and we have done this a number of times for pages with .NET. This is a workable scenario.
  2. For other back-end architectures, there are times when the AJAX code is a small standalone piece of code that can be "hooked" into either by redefining the function to add a hook to our code or using existing functionality to pass in our function. We did this a number of times in the past when AJAX was first being used by customers, but this is happening less and less over time1 because most pages are falling into the next category...
  3. For pages that use libraries - custom or open-source - some will have thousands of line of custom code with AJAX functionality built into it. These are often functions that cannot be modified and provide no hooks into the code. For these pages, we must resort to one of the following approaches.

1This entry was created in January of 2012 as more complicated and purposely-obfuscated code is being written by developers.

Approach #2 - Add event to parent container that exist at page load, then determine the event from there

The second approach has the advantage that you don't need any assistance from the customer. You simply need to identify the best parent container for adding your event(s).

In this example, we could attach an event to either the <form id="main-search" action="/Search"> node or the <div id="search" > node. We decided to attach it to the form since the search area is the "form", and it is "closer" to the nodes we want to add events to than the div is.

Here is the code we wrote to track this case: 

<script>
$opt={},$vopt={};
$opt.addEvent = function (el, evt, func, bubble) {
    if (!el || typeof (func) != "function") {
	return false;
    }
    bubble = bubble || false;
    var evtType = (window.attachEvent) ? "attach" : (window.addEventListener) ? "add" : "none";
    if (evtType == "attach") {
	el.attachEvent("on" + evt, func);
    } else if (evtType == "add") {
	el.addEventListener(evt, func, bubble);
    }
}
$opt.opCounters=[];
$opt.cntr = function(num,val){	
	var myPro = (("https:" == document.location.protocol) ? "https://by.essl.optimost.com/by" : "http://by.optimost.com");
	num = (typeof(num) != 'undefined') ? num : "";
	val = (typeof(val) != 'undefined') ? val : "";
	
	var randnum=Math.floor(Math.random()*100000);
	var url=myPro+"/counter/551/-/"+num+"/event.gif?op551gheaderliid={{$creative.sessionId}}&type="+val+"&session=" + randnum;
	$opt.opCounters[$opt.opCounters.length]=new Image();
	$opt.opCounters[$opt.opCounters.length-1].src=url;
	
}
$opt.enterKey = function(evt){

	if(evt.keyCode==13){
		//alert("counter fired");
		$opt.cntr(14,"unique");
	}
}
$opt.inputClick=function(evt){
	if(evt.keyCode!=13){
		if(evt.stopPropagation)evt.stopPropagation();
		else evt.cancelBubble=true;
	}
}
//Start Tracking code
$vopt.searchForm = document.getElementById("main-search");
$vopt.searchBarInput = $vopt.searchForm ? $opt.get("className","placeholder as-input","input",$vopt.searchForm):null;
$vopt.spanButton = $vopt.searchForm ? $opt.get("className","search replace","span",$vopt.searchForm):null;

//counters
$opt.addEvent($vopt.searchForm,'click',function(){$opt.cntr(14,"autoselect");});  //Autoselect returned <li> entry selected, click on entire form
$opt.addEvent($vopt.searchBarInput ,'keydown',$opt.enterKey);                     //User entered content and hit enter to select it
$opt.addEvent($vopt.spanButton,'click',function(){$opt.cntr(14,"unique");});      //User entered content and hit the button on the right to select it
$opt.addEvent($vopt.searchBarInput,'click',$opt.inputClick,false);                //An event is added so that a click in the text input is ignored
</script>

So to track the visitor's behavior, we need to account for the following contingencies:

  1. We need to fire an event when the visitor selects a search item. This means:
    • Fire a counter event when the visitor submits the text input in the search box that they populated.
    • Fire a counter event when the visitor selects one of the auto-populated entries returned via AJAX.
  2. Unfortunately, the javascript code added by the customer circumvents the normal submission process of the form and submits the form using javascript, so we cannot use the form's onsubmit event.

Given this, if we could magically add the events just after the auto-populated entries are returned, then we would add these events:

  1. A click event to the span magnifying glass to indicate the user selected their text input.
  2. A keydown event on the text input that looks for the enter key to be pressed by the user, indicating selection of their text input.
  3. A click event on the newly-created <ul class="as-list"> to indicate user selection of an auto-populated entry.

The issue is with the last event which cannot be added on page load since that node does not exist. So to solve this problem, we do the following:

  • We find the three nodes that exist in the original HTML on page load: The form node, the text input and the span containing the submit magnifying glass
  • We add the first two events above since those two nodes are available in the original HTML.
  • For the 3rd event, we add 2 events to the existing nodes:
    1. For the form node, we add a click event. This will capture when a visitor clicks on the autoselect entries that will later be added via AJAX. Unfortunately, it will also capture any click inside the form, some of which we don't want to track, so we also add...
    2. For the text input node, we then add a click event to cancel the propagation of that click if a user clicks inside the text input. A click inside the text input will be the visitor adding focus to the select input, BUT since it is inside the form, the form click we already added will also be triggered. So since the text input click will occur first - the form is a parent of the text input - we add an event to stop the event propogation, and now the form click event is not fired.

Approach #3 - Use a function from a library to detect AJAX requests

Some of the open-source libraries have functions that will indicate certain states in AJAX events. For example, jQuery has four functions to tell you about the progress of an AJAX transaction. The following code shows four useful state indicators: 

$(document).ajaxStart(function() { document.title+="start"; });
$(document).ajaxStop(function() { document.title+="stop"; });
$(document).ajaxComplete(function() { document.title+="Complete"; });
$(document).ajaxSuccess(function() { document.title+="Success"; });

In this simple example, the document.title will be modified to append the name of the jQuery AJAX Event that has fired. For the scenario in this WIKI entry, we could theoretically use ajaxStop or ajaxComplete or ajaxSuccess to make the necessary changes. If you test it, you will see that the normal order of triggering is "Start - Success - Complete - Stop".

Some points on Approach #3:

  1. In our list of scenarios where we need to add a "hook" to our code, the "page uses a library that has complicated and obfuscated code" scenario often involves jQuery as the underlying library. So this approach is an option with jQuery.
  2. If the test page uses jQuery, you need no assistance from the customer to track your events.
  3. For the example in this WIKI entry, it would allow you to add the 3rd event we had above after a succesful return of AJAX entries, while adding the first 2 events on page load as normal.
  4. For pages where you also need to make changes to the returned content - if the returned AJAX content is part of our MVT test - you can use this to trigger a execution of your module again.
