AJAX - Brief overview

As each day passes, more websites are using AJAX to efficiently pass data from servers to web pages without forcing a page refresh. This results in increased efficiency and interactivity, at the cost of increased page size (libraries usually included) and increased code complexity (cross-browser code needed to upload the content and render it)

AJAX and Optimost

Separation of data, format, style, and function

One of the advantages is that you can deliver information using a multitude of formats. This allows for smarter and more efficient applications. In terms of how this effect pages Optimost works on, this means that a page that is changing the colors and sizes available as the user scrolls through different products can send that information as:

  1. Raw data embedded in XML
  2. The actual HTML for the area to be replaced.

The first method is more efficient, but then you have to convert it to HTML to add it to the page. The second method is easier because you can just add it immediately to the page, but using innerHTML has some cross-browser issues since it is not a real DOM standard application. In most cases, our customers use method #2, and this has caused us some issues.

AJAX applications and innerHTML - What we need to look for

Most of our customers use AJAX to pass the HTML of the area that needs to be changed, and then add it to the innerHTML of the cell of interest. This works fine for our customers, but can be a problem for us since the elements may or may not be in the DOM. Here is what you need to look for (and I am assuming that we have a "hook" into their code so that as soon as the AJAX response is added to the page, our code gets called to make our changes):

  1. Find out how they are receiving the data and adding it to the page. Basically, where in the DOM is it being added. There may be multiple calls, find out for all of them.
  2. If you are moving content into or out of the area that is being written into the DOM, be very VERY careful. You may find that reading the content from the recently added area and/or adding content into the recently added area may fail. I had this problem in IE. Instead, I found an area of the page that was not being passed by the AJAX call, and did my DOM additions into there.

The Moral of the story? Understand how your page works before you implement any dynamic changes.

Specific example #1 - Product page with Ajax

A while back, we worked on a Product Page that loaded the product information usign Ajax. Our test included testing the contents within two of the areas that where passed using Ajax. So we needed the customer to "hook" us into the code so that as soon as the Ajax modifies the content via the DOM, we can step in and do our thing.

How it was done - adding the "hook"

So here is an example of how they added us into their code. There was an include named /shopping/js/fc_nav.js that contained the Ajax code. In there, they had the following: 

// AJAX call to reload divs.
function reloadDiv (pUrl, pDivId, pCallback) {
    var xmlhttp;
    // create the request
    xmlhttp = window.XMLHttpRequest?new XMLHttpRequest(): new ActiveXObject("Microsoft.XMLHTTP");
    // callback to update specified div
    xmlhttp.onreadystatechange = function () {
        if (xmlhttp.readyState == 4) {
            // alert ("x updating: " + pDivId + "\nURL: "+pUrl+"\nstatus: " + xmlhttp.status);
            document.getElementById(pDivId).innerHTML = xmlhttp.responseText;
            if (typeof(pCallback) == "function"){
            	pCallback();
            }
        }
    }
    // submit request
    xmlhttp.open("get", pUrl);
    xmlhttp.send(null);
}

This is a standard block that sets up the Ajax to make the call to pUrl, then await the response. Notice that the variable pCallback is passed in and, if it is function, it is executed after the response - the response happens to be HTML to replace some area - is written to the element with passed in id pDivId.

So we they could call reloadDiv as follows: 

reloadDiv(/catalog/productpurchasewrapper.jsp,"product-purchase");

But there is nothing passed as the 3rd paramenter, no function call. But then to hook us in, they could say: 

reloadDiv(/catalog/productpurchasewrapper.jsp,"product-purchase", opProductPurchaseReload);

and then we would put that function in our setup block: 

function opProductPurchaseReload(){
//redo variables impacted by Ajax changes

}

Specific example #2 - Passing data to Optimost directly using Ajax

We also had a case of a client wanting to test content that did not exist on the default version of the page. As we all know, we would need them to normally add it as hidden element, or as a javascript value. But they suggested passing it as Ajax. Like, how fun!

How it was done

So we start by making the Ajax call within out Setup block. 

//importXML() is called by the variables values that would need the Ajax response.
function importXML(){
	ajaxLoader("/bookonline/breaklistingsX.asp?BB=1&ht=All&mds=14&tp=WEB8A&lp=1&n=3&bpp=14");
}
function ajaxLoader(url){
	x=null; opIE=false;
	if (document.getElementById) {
		//var x = (window.ActiveXObject) ? new ActiveXObject('MSXML2.XMLHTTP.3.0'): new XMLHttpRequest();
		if (window.XMLHttpRequest) {// If IE7, Mozilla, Safari, and so on: Use native object.
			x= new XMLHttpRequest();
		}
		else{
			if (window.ActiveXObject) {// ...otherwise, use the ActiveX control for IE5.x and IE6.
				x= new ActiveXObject('MSXML2.XMLHTTP.3.0');
				opIE=true;
  			}
		}
	}
	if (x){
		x.onreadystatechange = function(){
			if (x.readyState == 4 && x.status == 200){
				if(!opIE) opParseXML(x.responseXML);
				else{
					ieResp = x.responseText;
					opParseXML(ieResp);

				}
			}
		}
		x.open("GET", url, true);
		x.send(null);
	}
}

So this setups the Ajax call, and when it successfully receives the response, it calls opParseXML, which parses the XML data - this is XML data being sent, not HTML wrapped in XML - as follows:

function opParseXML(allXML){
	var containerNode=null;

			}
		}
	}
	else {//IE if we use xmlOBJ.responseXML
		topNode=allXML.documentElement;
		if(topNode.nodeName==TOP_DATA_CONTAINER_NODE) containerNode=topNode;
	}
	//alert(topNode.firstChild.nodeName);
	if(containerNode!=null){
		var warnerBreakNodes = containerNode.getElementsByTagName("WarnerBreak");

			ents=warnerBreakNodes[i].getElementsByTagName("Entertainment");
			txts=warnerBreakNodes[i].getElementsByTagName("Text");
			for(var j=0;j<ents.length;j++) allNodes['WarnerBreak'+i+':Ents']=ents[j].firstChild.nodeValue;
			for(var j=0;j<txts.length;j++) allNodes['WarnerBreak'+i+':txts']=txts[j].firstChild.nodeValue;
			if ( warnerBreakNodes[i].attributes && warnerBreakNodes[i].attributes.length ) {
				for ( var k=0; k<warnerBreakNodes[i].attributes.length;k++ ) {
					var key = warnerBreakNodes[i].attributes[k].nodeName;
					if ( typeof(key) != "string" ) continue;

AJAX and jQuery

If the site has jQuery it might be possible to use jQuery(window).ajaxSuccess(function() { ... }); to call your changes when the page updates. When using it, ajaxSuccess will trigger an event whenever there has been a successful jQuery AJAX call on the page. You can then add callbacks to functions with your changes in, and it will perform the changes again once the AJAX call has been completed.
