Introduction

This entry is intended to show how one can integrate with SiteCatalyst, and show examples of scenarios where it requires no customer modifications to existing pages, and one that does require customer modifications of existing pages.

SiteCatalyst Integration

The SiteCatalyst code works as follows:

  1. The code defines an object - "s" and its necessary functions/properties/arrays/etc.
  2. Tracking parameters are attached to this object.
  3. Then the object is "fired" - the parameters are attached to a tracking pixel that logs the information on SiteCatalyst servers.

Pretty simple concept. Now, in order to integrate as a third-party, we must inject ourselves into the process AFTER step #1 but BEFORE step #3. Let's keep that in mind and look at two examples of how customers can setup the SiteCatalyst code in the HTML of their pages.

This code is located near the bottom of the customer's page - just above the </body> tag - and the code does the following:

  1. The first include defines the s object and all its necessary functionality
  2. The next block - javascript - add eVars and sprops to the object for the tracking
  3. The final block - also javascript - attaches an onload event to fire the tracking pixel that will transmit that information back to for tracking.

This customer integrated SiteCatalyst with consideration of adding eVars and sprops outside of this block of code, which can be done by the customer or third-parties on the page like Optimost.

Customer example - Optimost code in Tempate

So for this customer, we attach our eVars and sprops using the jQuery document.ready event, which occurs just before the onload event. The code below is inside an Optimost template. 

</script>
//Site Catalyst
$vopt.prop37="::test-bizhomep5-treatment";
$vopt.prop25="main:mktg:business::home:::test-bizhomep5-treatment";
$vopt.eVar25="main:mktg:business::home:::test-bizhomep5-treatment";
</script>
.....
.....
<script>
//passing eVar25, sprop25, sprop37 using YAHOO onDOMReady event.
$opt.addS=function(){
        if(typeof(s)=="object"){
	  nw= new Date();
                  s.prop20=nw.getTime();
                  s.prop37=$vopt.prop37;
                  s.prop25=$vopt.prop25;
                  s.eVar25=$vopt.eVar25;
        }
}

$(document).ready(function() {
  $opt.addS();
});
</script>

This works because the "s" object is defined at document.ready, but the tracking pixel has not been fired, so we can add values to "s" before the tracking event.

Alternate "no change" example - SiteCatalyst code loaded at the top of page, tracking pixel fired at bottom

The SiteCatalyst "best practices" are to load the code at the top of the page. Some customers do that, define the evars/sprops at the top, and then defer the tracking pixel until later in the page. In those instances, it is likely that our Trial Code and/or Module Code calls will happen "between" step #1 and #3. In that case, we can set the values in the template's setup block or module.

This code is also located near the bottom of the customer's page - just above the </body> tag - and the code does the following:

  1. The first include defines the s object and all its necessary functionality
  2. The next block - javascript - adds eVars and sprops to the object for the tracking
  3. At the bottom of that block, the tracking pixel is fired to transmit that information back to for tracking.

As we stated at the top, in order to integrate with SiteCatalyst, we need to add our values between step #1 and step #3. In this example, all the code is contiguous and executed in inline - no onload event to delay it - so there is no place for a third-party to integrate with SiteCatalyst.

How to integrate in this example - necessary modifications

So to integrate with this customer, the would have to make one of the following modifications:

  1. Add code to the include file hosted by the Customer - http://www.allstate.com/includes/s_code.js - or add an include file right after it that with our code in it. That code will contain a call to Optimost Code that will add the eVars/sprops.
  2. Modify the last piece of code - step #3 - to be fired as an onload, as done in the first example where we are properly integrated.
  3. Physically separate step #3 to be further down on the page, and allow the third-party to place code after step #2 but before step #3 in order to integrate. Putting our module code between #2 and #3 would work.
