Introduction

Attributes (and values) are a way to pass Optimost specific information about an Optimost "event" that could be useful for analytics purposes. An "event" here is defined as any time our servers are called, which could be one of three events:

  1. A Trial Code call
  2. A Counter Code call
  3. The firing of a clickthru

The Trial Code and Counter Code automatically send back:

  1. A Trial Code call (with its Page code)
    All Query string parameters in top.location and all first-party cookies that start with "op"
  2. A Counter Code call
    All Query string parameters in top.location that start with "op" and all first-party cookies that start with "op"

Naming Conventions

When assigning names to attribute values, please do not use quotation marks in the name. When the analytics team needs to download reports for offline analysis, the extra quotation marks ruin the column formatting of the downloaded CSV files.

Example

In this case, the customer supplied Optimost with a list of values for a query string parameter in the page URL "sendroicid" that correspond to different traffic sources on the page under test (where the Trial code is). The values are 32-digit Globally Unique Identifiers (GUID), and some sources have a single values, others can have up to 14. Now, since this is a page with Trial code, it automatically gets all query string parameters, so we have access to all the GUIDs. However, they are very cryptic looking entries, and since some traffic sources may have different numbers of GUIDs, it would make reporting much easier if we sent back just the traffic source. So to that end, I wrote custom code to parse it out.

I stored the GUIDs in an object called iiSources, and created an Array for each source with multiple values. I then compared the incoming contents of the query string parameter "sendroicid", and if I get a match, I then pass the source name back to Optimost.

The way I pass back the value is by creating the optrial object, and adding an element "opsource" with the value set to the name of the traffic source (the name "opsource" was my choice, you just need to make sure its unique). Optimost has set the Trial code to look for the optrial object, and if it exists, to add it's elements to the list of name-value pairs to be passed back to Optimost via the query string of the trial code call back to Optimost. On the Optimost end, at the subject level, under page view attributes, you would set an attribute to look for this value. Name it Source, and say it is a javascript variable with then name "optrial.opsource", and save it.

Note that the same thing can be done with counters, only we use the opcounter object, and replace optrial with opcounter in the code we use. With both, make sure the customer adds the javascript code that captures the attributes ABOVE the trial or counter code.

<script>
var query  = unescape(top.location.search.substring(1)).split('&');
var params = new Object;
for (i = 0; i < query.length; i++){
	var tmp = query[i].split('=');
	params[tmp[0]] = tmp[1];
}
match=null;
if(params['sendroicid']!=null){
	for(var n in iiSources){
		if(typeof(iiSources[n])=='object'){// This means it is an array, so there are multiple values for this sources
			for(var j=0;j<iiSources[n].length;j++){
				if(iiSources[n][j]==params['sendroicid']) match=n;
			}
		}
		else if(iiSources[n]==params['sendroicid']) match=n;  //This means this source has a single value.
	}
}
//alert(match);
if(match!=null){
	if(typeof(optrial)!="object"){
		optrial=new Object;
		optrial.opsource=match;
	}
	else optrial.opsource=match;
}
if(typeof(optrial)=="object"){
	list='';
	for(var n in optrial) list+=n+' has the value '+optrial[n]+'\n';
	alert(list);
}
</script>

If things aren't showing up in the reports

Chances are you'll need to reset the wave and refresh creatives to fix it, this has happened on many occasions and resetting the wave usually fixes it
