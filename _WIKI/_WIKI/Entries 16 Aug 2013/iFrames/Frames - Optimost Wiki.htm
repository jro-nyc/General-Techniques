Frames - Brief overview

HTML frames allow authors to present documents in multiple views, which may be independent windows or subwindows. Multiple views offer designers a way to keep certain information visible, while other views are scrolled or replaced. For example, within the same window, one frame might display a static banner, a second a navigation menu, and a third the main document that can be scrolled through or replaced by navigating in the second frame.

Here is a simple frame document: 

<!DOCTYPE HTML PUBLIC "-//W3C//DTD HTML 4.01 Frameset//EN"
   "http://www.w3.org/TR/html4/frameset.dtd">
<HTML>
<HEAD>
<TITLE>A simple frameset document</TITLE>
</HEAD>
<FRAMESET cols="20%, 80%">
  <FRAMESET rows="100, 200">
      <FRAME src="contents_of_frame1.html">
      <FRAME src="contents_of_frame2.gif">
  </FRAMESET>
  <FRAME src="contents_of_frame3.html">
  <NOFRAMES>
      <P>This frameset document contains:
      <UL>
         <LI><A href="contents_of_frame1.html">Some neat contents</A>
         <LI><IMG src="contents_of_frame2.gif" alt="A neat image">
         <LI><A href="contents_of_frame3.html">Some other neat contents</A>
      </UL>
  </NOFRAMES>
</FRAMESET>
</HTML>

that might create a frame layout something like this:

Frames and Optimost

You are not likely to run into Frames-based pages we are Optimizing. But if you do, a few issues need to concern you.

Our Experiment and the Frames

In the example above, the Frame document page contains 3 frames, and of those frames, 2 contain HTML pages of interest and one contains just an image. So if we are running an experiment, we need to make sure the customer understands the Page with our code on it is NOT the Frame document, but in one of the Frames, ie the URL that the src attributes in the <FRAME > tag points to.

In this example, let's say we are testing the first frame, that frame is <FRAME src="contents_of_frame1.html"> so our code would live in the contents_of_frame1.html file on their servers.

Sharing data among frames

As mentioned above, insist we put all our code on the target Frame file. But what if this is an issue for our Universal code - Page Code in Modular templates or Page Code and Flash Code in Flash XML templates? Theoretically, a customer could propogate the Universal code throughout a bunch of Frame documents in their site, but cannot put the tags into the Frame container. (This issue is more likely in Flash XML templates). What to do?

We do not have direct visibility into the Frame document page from the different frames - access to the parent frame. But if we access DOM elements or javascript functions trough the parent object, we can do it. So if we have the Page Code and Flash code in the Frame document page and the Flash that calls optimost.S and optimost.Paths is in one of the Frames, then you need to call them as parent.optimost.S and parent.optimost.Paths


Example that access the content of Parent Frame

Click on the link below and see a Frame document page with 2 Frames, and the Optimost Page Code and Flash Code in the Frame document page. The alerts show you how we access the Optimost function defined in the base container HTML file
http://kidwww.optimost.com/webtest/QA/GM/Frames.html?myQS=outerFrameDocument

You will see from the alerts that we can access the Optimost object from the Frame document, but not from the frame, but from the frame we can access it as parent.Optimost.

Note that......

In the Frame document, the URL is:http://kidwww.optimost.com/webtest/QA/GM/Frames.html?myQS=outerFrameDocument and for the frame is it vehicle_index.html?myQSinner=innerFrame. One has the query string parameter myQS and the other has the query string parameter myQSinner. You will notice that in the alerts, myQS shows up, not myQSinner. However, in the inside Frame, I added Page Code and Flash code, but changed the name to optimost2, and I am also alerting the output of those functions, and you will see that optimost2 sees myQSinner.

I trust the reasons for this behavior make sense.
